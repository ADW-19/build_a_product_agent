# 第二章：长期记忆与短期记忆

> **核心论点**：短期记忆让模型能"记住刚才说了什么"，长期记忆让模型能"记住你是一个什么样的人"。本章不讨论心理学概念，只讲工程实现——怎么让 Agent 在每一次对话中准确判断"什么该存入 Milvus 长期记住"、"什么只是本次对话的上下文"、以及当记忆太多时如何防止上下文爆炸。

---

## 2.1 概念澄清：工程意义上的长短期记忆

### 2.1.1 类比：人类怎么记东西

你和一个同事聊天：

- **短期记忆**：刚才 5 分钟你们聊了什么——"他说下周三要出差"、"我答应帮他查一下数据"。这些信息只在这次对话中需要，对话结束就可以忘掉。
- **长期记忆**：你认识这个同事三年了——"他叫张三，是后端工程师，擅长 Python，有两个孩子，对芒果过敏，上次一起吃饭是三个月前"。这些信息跨越多场对话，应该持久记住。

**对 Agent 来说，工程上的定义非常朴素：**

| 维度 | 短期记忆 | 长期记忆 |
|------|---------|---------|
| 生命周期 | 当前会话内有效，会话结束可丢弃 | 跨会话持久保留 |
| 存储位置 | Redis（快，可过期） | Milvus / 向量数据库（持久化，语义检索） |
| 数据形式 | 消息列表 `[{role, content}, ...]` | 结构化记忆条目 + 向量 embedding |
| 检索方式 | 直接按 session_id 全量取出 | 按语义相似度检索最相关的 N 条 |
| 典型内容 | "帮我查一下今天的天气" → "好的，今天晴天" | "用户叫张三，住在北京，喜欢简洁的回答" |
| 容量 | 受限（几十轮对话后裁剪） | 几乎无限（向量库可存百万条） |

### 2.1.2 为什么不是"给模型一个大 context 就行了"

有人问：模型上下文窗口已经到 128K token 了，为什么还要分长短期？直接把所有历史记录全塞进 prompt 不就行了？

三个原因：

1. **成本**：每轮对话都塞入 10K token 的历史，一天 1000 次对话 = 1000 万 token 的上下文处理。按 GPT-4o 输入 $2.50/1M token 算，每天多花 $25，一年 $9,000——全花在重复处理旧信息上。
2. **注意力衰减**：模型在长上下文中对中间位置的信息注意力显著下降（Lost in the Middle 现象）。塞了 50 条旧记忆，真正关键的那条可能被淹没。
3. **响应变慢**：上下文越长，首 token 延迟越高。128K 的 prompt 首 token 延迟可能是 1K 的 3~5 倍。

**结论：长短期记忆分离不是为了省事，是为了省钱、省时间、提准确率。**

## 2.2 短期记忆：会话内上下文管理

### 2.2.1 短期记忆的本质

短期记忆就是当前 session 的对话历史。上一章 1.2 节已经在 Redis 中按 `session_id` 隔离实现了。这里重点补充两个工程问题：**消息裁剪策略**和**多轮对话的 token 计算**。

### 2.2.2 消息裁剪：不能简单地"保留最近 N 条"

```python
# ❌ 粗暴裁剪——可能裁掉关键信息
messages = messages[-MAX_HISTORY_LENGTH:]  # 保留最近 50 条
```

问题：用户在对话开头说了一句关键信息（如"用中文回复"），50 轮后被裁掉了，Agent 突然开始说英文。

**改进：分层裁剪——系统消息永不裁剪，用户长消息做摘要替换。**

```python
# core/short_memory.py
from typing import Literal

MAX_TOTAL_TOKENS = 8000       # 历史消息总 token 上限（留出空间给模型回复）
MAX_SINGLE_MSG_TOKENS = 2000  # 单条消息超过此值，替换为摘要
KEEP_SYSTEM_MESSAGES = True   # 系统消息永不裁剪


async def trim_history(
    messages: list[dict],
    max_tokens: int = MAX_TOTAL_TOKENS,
) -> list[dict]:
    """
    智能裁剪对话历史。

    规则：
    1. SystemMessage 永不删除
    2. 从最早的消息开始删除（先进先出）
    3. 但保留最近 4 条（最少保留轮数，保证对话连贯性）
    4. 长消息优先替换为摘要而不是直接删除
    """
    if not messages:
        return messages

    # 分离系统消息
    system_msgs = [m for m in messages if m.get("role") == "system"]
    dialog_msgs = [m for m in messages if m.get("role") != "system"]

    total_tokens = _count_tokens(messages)

    if total_tokens <= max_tokens:
        return messages

    # 保留最近 4 条对话消息（最少保留轮数）
    MIN_KEEP = 4
    keep_tail = dialog_msgs[-MIN_KEEP:] if len(dialog_msgs) >= MIN_KEEP else dialog_msgs
    trim_pool = dialog_msgs[:-MIN_KEEP] if len(dialog_msgs) > MIN_KEEP else []

    # 从头（最旧）开始删除，直到 token 数达标
    result = list(trim_pool)
    while _count_tokens(system_msgs + result + keep_tail) > max_tokens and result:
        removed = result.pop(0)  # 删最早的消息

    return system_msgs + result + keep_tail


def _count_tokens(messages: list[dict]) -> int:
    """估算消息列表的 token 数（近似值，用字符数 / 2 粗略估算中英文混合场景）"""
    total_chars = sum(len(m.get("content", "")) for m in messages)
    return total_chars // 2  # 中英文混合，约 2 字符 ≈ 1 token
```

### 2.2.3 Token 精确计数（生产环境）

上面的 `//2` 是粗略估算。生产环境应该用模型对应的 tokenizer 精确计算：

```python
# core/token_counter.py
import tiktoken

# GPT-4o / GPT-4 使用的编码
ENCODING = tiktoken.get_encoding("o200k_base")


def count_tokens(text: str) -> int:
    """精确计算文本的 token 数"""
    return len(ENCODING.encode(text))


def count_message_tokens(messages: list[dict]) -> int:
    """
    计算消息列表的 token 数。
    包含每条消息的角色标识和分隔符的开销（约 4 token/条）。
    """
    total = 0
    for msg in messages:
        total += count_tokens(msg.get("content", ""))
        total += 4  # 消息格式开销（role + 分隔符）
    return total
```

### 2.2.4 短期记忆的完整读取流程

```python
# services/chat_service.py
from core.session import get_history, append_message_with_limit
from core.short_memory import trim_history

async def prepare_context(session_id: str, user_message: str) -> list[dict]:
    """
    为 Agent 准备上下文消息列表。

    组装顺序：系统提示词 → 长期记忆（检索） → 短期记忆（会话历史） → 当前用户消息
    """
    messages = []

    # 1. 系统提示词（角色设定）
    messages.append({
        "role": "system",
        "content": "你是用户的 AI 助手。回答简洁、准确。用中文回复。",
    })

    # 2. 长期记忆（从 Milvus 检索，见 2.3 节）
    long_term_memories = await retrieve_long_term_memories(session_id, user_message)
    if long_term_memories:
        memory_text = "以下是关于此用户的已知信息：\n" + "\n".join(
            f"- {m['content']}" for m in long_term_memories
        )
        messages.append({"role": "system", "content": memory_text})

    # 3. 短期记忆（本次会话的历史）
    history = await get_history(session_id)
    history = await trim_history(history)  # 裁剪
    messages.extend(history)

    # 4. 当前用户消息
    messages.append({"role": "user", "content": user_message})

    return messages
```

## 2.3 长期记忆：跨会话持久化到 Milvus

### 2.3.1 什么是"长期记忆"——Agent 视角的工程定义

长期记忆不是简单的"把对话记录存下来以后搜"。它的核心是**从对话中提取出关于用户的事实、偏好、知识，以结构化的形式持久存储，并在未来的对话中按语义相关性检索回来**。

举个例子：

```
对话 1（周一）：
  用户：我最近在学 Rust，太折磨了
  Agent：Rust 的学习曲线确实陡峭，需要帮忙吗？
  → 提取长期记忆：{"fact": "用户正在学习 Rust 语言", "category": "学习", "confidence": 0.9}

对话 2（周三）：
  用户：有没有什么推荐的编程书？
  Agent：（检索到长期记忆"用户正在学习 Rust"）
        → 推荐《The Rust Programming Language》而不是通用的《代码大全》
```

如果没有从对话 1 中提取并存储"用户正在学 Rust"这个事实，对话 2 中的 Agent 只会泛泛推荐。有了长期记忆，Agent 的回答从"通用"变成了"个性化"。

### 2.3.2 存储架构：Redis 做短期，Milvus 做长期

### 2.3.3 Milvus Collection 设计

```python
# core/long_memory_store.py
from pymilvus import (
    connections, Collection, CollectionSchema,
    FieldSchema, DataType, utility,
)

# 连接 Milvus
connections.connect(host="localhost", port="19530")

# 定义 Collection Schema
LONG_TERM_MEMORY_SCHEMA = CollectionSchema([
    FieldSchema(name="id", dtype=DataType.INT64, is_primary=True, auto_id=True),
    FieldSchema(name="session_id", dtype=DataType.VARCHAR, max_length=64),
    FieldSchema(name="user_id", dtype=DataType.VARCHAR, max_length=64),
    FieldSchema(name="content", dtype=DataType.VARCHAR, max_length=2000),
    FieldSchema(name="category", dtype=DataType.VARCHAR, max_length=32),
    FieldSchema(name="importance", dtype=DataType.INT64),    # 1~10，重要程度
    FieldSchema(name="confidence", dtype=DataType.FLOAT),    # 0~1，LLM 判断的可信度
    FieldSchema(name="embedding", dtype=DataType.FLOAT_VECTOR, dim=1536),
    FieldSchema(name="created_at", dtype=DataType.INT64),
    FieldSchema(name="updated_at", dtype=DataType.INT64),
    FieldSchema(name="access_count", dtype=DataType.INT64),  # 被检索到的次数
])

# 创建 Collection
if not utility.has_collection("long_term_memories"):
    collection = Collection(name="long_term_memories", schema=LONG_TERM_MEMORY_SCHEMA)

    # 为 embedding 向量建 IVF_FLAT 索引
    collection.create_index(
        field_name="embedding",
        index_params={
            "metric_type": "IP",       # 内积相似度（向量已归一化时等同于余弦相似度）
            "index_type": "IVF_FLAT",
            "params": {"nlist": 128},
        },
    )
    collection.load()
```

关键字段说明：

| 字段 | 作用 | 为什么需要 |
|------|------|-----------|
| `content` | 记忆内容文本 | 检索回来后直接塞进 prompt |
| `category` | 分类标签 | 支持按类别过滤（个人资料 / 偏好 / 知识 / 待办） |
| `importance` | 重要程度 1~10 | 检索时加权，重要记忆优先返回 |
| `confidence` | LLM 判断的可信度 0~1 | 低置信度的记忆可以忽略或降权 |
| `access_count` | 访问次数 | 识别"僵尸记忆"（存了但从没用过，可以清理） |
| `updated_at` | 最后更新时间 | 同一条记忆被多次提及，更新时间但保留最早创建时间 |

### 2.3.4 核心难题：怎么让模型准确判断"什么该存入长期记忆"

这是整个系统的关键。你不能把每一句对话都存进 Milvus，否则库里 90% 是噪音——"好的""谢谢""知道了"这些毫无长期价值的对话会把真正的记忆淹没。

**方案：在每次对话结束后，让 LLM 做一次"记忆提取审查"。**

```python
# core/memory_extractor.py
import json
import time
from typing import Optional
from pydantic import BaseModel, Field
from langchain_core.prompts import ChatPromptTemplate
from core.llm import call_llm_with_retry
from core.long_memory_store import insert_memories
from core.logger import get_logger

logger = get_logger(__name__)


# ============ 第一步：定义记忆提取的输出结构 ============

class MemoryItem(BaseModel):
    """单条长期记忆的结构化定义"""
    content: str = Field(..., description="记忆内容。用完整句子表述，必须包含主语。例如'用户正在学习 Rust'而不是'学 Rust'")
    category: str = Field(..., description="分类：personal(个人资料) / preference(偏好习惯) / knowledge(知识技能) / event(事件经历) / todo(待办事项) / other")
    importance: int = Field(..., ge=1, le=10, description="重要程度。10=核心身份信息，1=可能有用但不确定")
    confidence: float = Field(..., ge=0.0, le=1.0, description="可信度。1.0=用户明确说出的事实，0.5=从上下文推断的")


class MemoryExtractionResult(BaseModel):
    """记忆提取的完整结果"""
    new_memories: list[MemoryItem] = Field(default_factory=list)
    update_memories: list[dict] = Field(default_factory=list)  # [{"old_content": "...", "new_content": "..."}]
    delete_memories: list[str] = Field(default_factory=list)   # 需要删除的记忆内容
    notes: str = Field(default="", description="提取推理说明")


# ============ 第二步：构造记忆提取的 prompt ============

MEMORY_EXTRACTION_PROMPT = ChatPromptTemplate.from_messages([
    ("system", """你是一个记忆提取系统。你的任务是从对话中提取关于用户的长期记忆。

## 什么是值得存入长期记忆的信息：
1. **个人资料**：姓名、年龄、职业、所在地、家庭成员、联系方式
2. **偏好习惯**：喜欢/讨厌什么、沟通风格偏好（简洁/详细）、常用语言
3. **知识技能**：掌握的技术、学习进度、专业领域
4. **事件经历**：做过的事、去过的地方、重要日期
5. **待办事项**：用户提到的计划、承诺、截止日期

## 什么不应该存入长期记忆：
- 一次性的简单问答（"今天天气怎么样"——这是每次都要查的，不是记忆）
- 闲聊寒暄（"你好""谢谢""再见"）
- 纯粹的知识性内容（"Python 是什么"——这是知识库的事，不是用户记忆）
- 临时的会话状态（"刚才我说了A"——短期记忆就够了）
- 模糊不确定的推断（不确定就不要存）

## 记忆更新规则：
- 如果用户提供了与已有记忆**矛盾**的信息，标记为 update（新信息覆盖旧信息）
- 如果用户提供的只是**补充**（不矛盾），新增一条独立记忆
- 如果用户明确否认之前的信息（"不对，我不是..."），标记为 delete

## 已有的长期记忆（避免重复存储）：
{existing_memories}

## 重要程度评分标准：
- 10：核心身份信息（姓名、职业、重要关系）
- 7-9：重要偏好和习惯、正在学习的关键技能
- 4-6：有价值的参考信息、一般事件
- 1-3：边缘信息、可能有用但不重要

## 输出要求：
只输出 JSON，不要任何解释文字。严格遵循指定格式。
"""),
    ("user", """请分析以下对话，提取值得存入长期记忆的信息。

对话内容：
{dialogue}
"""),
])


# ============ 第三步：执行记忆提取 ============

async def extract_long_term_memories(
    session_id: str,
    dialogue: list[dict],
    existing_memories: list[str],
) -> MemoryExtractionResult:
    """
    从对话中提取长期记忆。

    每次对话结束后调用此函数，让 LLM 判断对话中是否有值得长期记住的信息。
    """
    dialogue_text = "\n".join(
        f"{m['role']}: {m['content']}" for m in dialogue[-10:]  # 只要最近 10 条
    )

    prompt = MEMORY_EXTRACTION_PROMPT.format(
        existing_memories="\n".join(f"- {m}" for m in existing_memories) if existing_memories else "（暂无已有记忆）",
        dialogue=dialogue_text,
    )

    try:
        # 用一个小模型做记忆提取，省成本
        response = await call_llm_with_retry(
            messages=prompt.to_messages(),
            model="gpt-4o-mini",  # 记忆提取不需要大模型，4o-mini 足够
            response_format={"type": "json_object"},  # 强制 JSON 输出
        )

        result = MemoryExtractionResult(**json.loads(response))

        if result.new_memories:
            logger.info(
                "提取到 %d 条新记忆 | session_id=%s | categories=%s",
                len(result.new_memories),
                session_id,
                [m.category for m in result.new_memories],
            )
        else:
            logger.debug("本轮对话无新的长期记忆 | session_id=%s", session_id)

        return result

    except Exception as e:
        logger.warning("记忆提取失败 | session_id=%s | error=%s", session_id, str(e))
        return MemoryExtractionResult()  # 提取失败不阻塞对话，返回空


# ============ 第四步：将提取的记忆写入 Milvus ============

async def persist_memories(
    session_id: str,
    user_id: str,
    extraction_result: MemoryExtractionResult,
) -> None:
    """
    将 LLM 提取的记忆写入 Milvus。

    每一条记忆需要先生成 embedding 向量，再插入 Milvus。
    """
    now = int(time.time())

    for memory in extraction_result.new_memories:
        # 生成 embedding
        embedding = await _embed_text(memory.content)

        await insert_memory(
            session_id=session_id,
            user_id=user_id,
            content=memory.content,
            category=memory.category,
            importance=memory.importance,
            confidence=memory.confidence,
            embedding=embedding,
            created_at=now,
            updated_at=now,
        )

    # 处理更新
    for update in extraction_result.update_memories:
        await update_memory_content(update["old_content"], update["new_content"])

    # 处理删除
    for content in extraction_result.delete_memories:
        await delete_memory_by_content(content)
```

### 2.3.5 让 LLM 输出结构化 JSON 的关键

上面 `call_llm_with_retry` 调用了 `response_format={"type": "json_object"}`。这是 OpenAI 的 JSON Mode，确保 LLM 输出合法 JSON。但你还需要配合 Pydantic 做二次校验：

```python
# 即使开了 JSON Mode，LLM 输出的 JSON 结构也可能不完美
# 所以用 Pydantic 做二次校验
try:
    raw = json.loads(response)
    result = MemoryExtractionResult(**raw)
except (json.JSONDecodeError, ValidationError) as e:
    logger.warning("记忆提取 JSON 解析失败 | response=%s", response[:200])
    return MemoryExtractionResult()  # 解析失败返回空，不阻塞对话
```

### 2.3.6 调用时机：什么时候触发记忆提取

**不要在同步的请求-响应链路中做记忆提取。** 因为：
- LLM 调用需要 1~3 秒，会拖慢用户感知的回复速度
- 记忆提取失败了也不应该影响正常对话

**正确做法：对话返回给用户后，异步后台执行。**

```python
import asyncio
from core.memory_extractor import extract_long_term_memories, persist_memories
from core.long_memory_store import search_similar_memories

# routes/chat.py 中的调用方式
@router.post("/completions")
async def chat_completions(request: ChatRequest):
```

### 2.3.7 长期记忆的检索：语义相似度查询

```python
# core/long_memory_store.py（续）
from pymilvus import Collection


async def search_similar_memories(
    user_id: str,
    query_text: str,
    top_k: int = 5,
    min_similarity: float = 0.7,
) -> list[dict]:
    """
    根据当前对话内容，检索最相关的长期记忆。

    检索过程：
    1. 将当前用户消息转为 embedding 向量
    2. 在 Milvus 中搜索最相似的记忆
    3. 过滤掉相似度过低的结果
    4. 按 importance × similarity 加权排序
    """
    collection = Collection("long_term_memories")

    # 1. 生成查询向量
    query_embedding = await _embed_text(query_text)

    # 2. 向量相似度搜索
    results = collection.search(
        data=[query_embedding],
        anns_field="embedding",
        param={"metric_type": "IP", "params": {"nprobe": 16}},
        limit=top_k * 2,  # 多取一些，后面再过滤
        expr=f'user_id == "{user_id}"',  # 只搜当前用户
        output_fields=["content", "category", "importance", "confidence", "access_count"],
    )

    # 3. 过滤 + 加权
    scored_memories = []
    for hit in results[0]:
        similarity = hit.score  # 内积相似度
        if similarity < min_similarity:
            continue

        entity = hit.entity
        # 加权 = 相似度(0.7) + 重要程度(0.2) + 可信度(0.1)
        weighted_score = (
            similarity * 0.7 +
            (entity.importance / 10) * 0.2 +
            entity.confidence * 0.1
        )

        scored_memories.append({
            "content": entity.content,
            "category": entity.category,
            "importance": entity.importance,
            "confidence": entity.confidence,
            "similarity": similarity,
            "weighted_score": weighted_score,
        })

    # 按加权分数降序
    scored_memories.sort(key=lambda x: x["weighted_score"], reverse=True)
    selected = scored_memories[:top_k]

    # 4. 更新访问计数（异步，不阻塞返回）
    for mem in selected:
        asyncio.create_task(_increment_access_count(user_id, mem["content"]))

    logger.debug("长期记忆检索 | query=%s | found=%d", query_text[:50], len(selected))
    return selected
```

## 2.4 上下文爆炸问题

### 2.4.1 "上下文爆炸"是什么意思

Agent 每次对话需要把以下内容塞进 prompt：

```
┌──────────────────────────────────────────┐
│ 系统提示词                                │  ~200 token
│ 长期记忆（检索到的）                       │  ~500 token（正常）
│ 短期记忆（会话历史）                       │  ~2000 token（正常）
│ 工具定义                                  │  ~300 token
│ 当前用户消息                              │  ~100 token
│                                          │
│ 合计                                      │  ~3100 token ✅ 很健康
└──────────────────────────────────────────┘
```

但如果：
- 长期记忆检索了 20 条（4000 token）
- 用户连续聊了 100 轮（10000 token）
- 工具定义复杂（1000 token）

```
┌──────────────────────────────────────────┐
│ 系统提示词                                │  ~200 token
│ 长期记忆（20条，未过滤）                   │  ~4000 token
│ 短期记忆（100轮历史）                      │  ~10000 token
│ 工具定义                                  │  ~1000 token
│ 当前用户消息                              │  ~100 token
│                                          │
│ 合计                                      │  ~15300 token ⚠️ 逼近上限
└──────────────────────────────────────────┘
```

这就是上下文爆炸——太多信息塞进 prompt，导致：
1. **超出模型上下文窗口** → API 直接报错
2. **注意力稀释** → 模型忽略关键信息
3. **成本飙升** → 每轮对话都在为无用的历史 token 付费
4. **延迟增加** → prompt 越长，首 token 越慢

### 2.4.2 爆炸源一：长期记忆检索过多

**解法：严格控制检索数量 + 相似度阈值**

```python
# ❌ 检索 20 条全塞进去
memories = await search_similar_memories(user_id, query, top_k=20)

# ✅ 分层过滤，只保留高价值的
MAX_LTM_TOKENS = 800  # 长期记忆最多占 800 token
MIN_WEIGHTED_SCORE = 0.65  # 加权分数低于此值不要


async def retrieve_and_filter_memories(
    user_id: str,
    query_text: str,
    max_tokens: int = MAX_LTM_TOKENS,
) -> list[dict]:
    """检索长期记忆并做严格过滤"""
    # 第一步：候选检索
    candidates = await search_similar_memories(
        user_id=user_id,
        query_text=query_text,
        top_k=15,
        min_similarity=0.65,
    )

    # 第二步：按加权分排序，逐个加入，直到接近 token 上限
    selected = []
    token_count = 0
    for mem in candidates:
        if mem["weighted_score"] < MIN_WEIGHTED_SCORE:
            continue

        mem_tokens = len(mem["content"]) // 2
        if token_count + mem_tokens > max_tokens:
            break

        selected.append(mem)
        token_count += mem_tokens

    # 第三步：如果高重要度记忆被遗漏，提升它们
    high_importance = [m for m in candidates if m["importance"] >= 8 and m not in selected]
    for mem in high_importance[:3]:  # 最多额外加 3 条高重要度记忆
        selected.append(mem)

    return selected
```

### 2.4.3 爆炸源二：短期历史过长

**解法：滚动摘要——把旧对话压缩为摘要，保留最近对话的完整内容。**

```python
# core/context_compressor.py
from langchain_core.messages import HumanMessage, AIMessage
from core.llm import call_llm_with_retry
from core.logger import get_logger

logger = get_logger(__name__)


async def compress_conversation(
    messages: list[dict],
    keep_recent: int = 6,       # 保留最近 N 条完整消息
    max_summary_tokens: int = 500,
) -> list[dict]:
    """
    对话压缩：旧消息 → 摘要 + 最近消息保持完整。

    结果：
    [
        {"role": "system", "content": "[对话摘要] 用户询问了天气..."},
        {"role": "user", "content": "最近的第 3 轮消息"},   ← 完整保留
        {"role": "assistant", "content": "最近的第 2 轮消息"}, ← 完整保留
        {"role": "user", "content": "最新消息"},             ← 完整保留
    ]
    """
    if len(messages) <= keep_recent:
        return messages

    # 需要压缩的部分
    old_messages = messages[:-keep_recent]
    recent_messages = messages[-keep_recent:]

    # 如果旧消息太少，不值得压缩
    if len(old_messages) <= 2:
        return messages

    # 生成摘要
    dialogue_text = "\n".join(
        f"{m['role']}: {m['content'][:200]}" for m in old_messages
    )

    summary_prompt = [
        {"role": "system", "content": "用不超过 200 字总结以下对话的关键信息。只记录事实和决定，不记录寒暄。保留用户提到的所有人名、数字、日期、承诺和偏好。"},
        {"role": "user", "content": dialogue_text},
    ]

    try:
        summary = await call_llm_with_retry(
            messages=summary_prompt,
            model="gpt-4o-mini",
        )
        logger.info("对话压缩完成 | 原始 %d 条 → 摘要 %d 字", len(old_messages), len(summary))
    except Exception:
        logger.warning("对话压缩失败，使用最后 %d 条原始消息", keep_recent)
        return recent_messages

    # 组装结果
    result = [{"role": "system", "content": f"[历史对话摘要] {summary}"}]
    result.extend(recent_messages)

    return result
```

### 2.4.4 爆炸源三：工具定义的描述过长

工具定义的描述（docstring）会被转成 function calling 的 schema 发给 LLM。如果每个工具的 docstring 都是一个小作文，10 个工具就能吃掉 2000 token。

**解法：工具描述只写"做什么"和"参数含义"，不写"怎么做"。**

### 2.4.5 终极方案：上下文预算管理系统

把上述各种策略整合为一个统一的上下文预算管理器：

```python
# core/context_budget.py
from dataclasses import dataclass, field
from core.token_counter import count_message_tokens


@dataclass
class ContextBudget:
    """
    上下文预算管理器。

    每条请求的总 token 预算 = 模型上下文窗口 - 回复预留空间。
    各组件按优先级分配 token 份额。
    """
    model_max_tokens: int = 128000       # 模型最大上下文
    response_reserve: int = 4096          # 留给模型回复的 token
    total_budget: int = field(init=False)

    # 各组件份额分配（总预算 = total_budget）
    system_prompt_share: float = 0.05     # 系统提示词：5%
    long_term_memory_share: float = 0.10  # 长期记忆：10%
    short_term_memory_share: float = 0.75 # 短期记忆：75%（大头）
    tool_def_share: float = 0.05          # 工具定义：5%
    current_message_share: float = 0.05   # 当前消息：5%

    def __post_init__(self):
        self.total_budget = self.model_max_tokens - self.response_reserve

    @property
    def long_term_memory_limit(self) -> int:
        return int(self.total_budget * self.long_term_memory_share)

    @property
    def short_term_memory_limit(self) -> int:
        return int(self.total_budget * self.short_term_memory_share)


# 使用示例
budget = ContextBudget(model_max_tokens=128000)

async def build_context(request, budget: ContextBudget) -> list[dict]:
    """在预算约束下组装上下文"""
    messages = []

    # 1. 系统提示词（固定，通常不超预算）
    messages.append({"role": "system", "content": SYSTEM_PROMPT})

    # 2. 长期记忆（预算：10%）
    memories = await retrieve_and_filter_memories(
        request.user_id,
        request.message,
        max_tokens=budget.long_term_memory_limit,
    )
    if memories:
        messages.append({"role": "system", "content": _format_memories(memories)})

    # 3. 短期记忆（预算：75%，但需要压缩）
    history = await get_history(request.session_id)
    
    # 如果历史 token 超预算，触发压缩
    history_tokens = count_message_tokens(history)
    if history_tokens > budget.short_term_memory_limit:
        logger.info("历史过长触发压缩 | before=%d tokens", history_tokens)
        history = await compress_conversation(
            history,
            keep_recent=6,
            max_summary_tokens=budget.short_term_memory_limit // 2,
        )
        logger.info("压缩后 | after=%d tokens", count_message_tokens(history))

    messages.extend(history)

    # 4. 当前消息
    messages.append({"role": "user", "content": request.message})

    total = count_message_tokens(messages)
    logger.info("上下文组装完成 | total_tokens=%d / budget=%d", total, budget.total_budget)

    return messages
```

## 2.7 本章小结

| 概念 | 存储 | 生命周期 | 检索方式 | 容量管理 |
|------|------|---------|---------|---------|
| 短期记忆 | Redis | 当前会话 | 按 session_id 全量取出 | token 裁剪 + 滚动摘要压缩 |
| 长期记忆 | Milvus | 跨会话持久 | 语义相似度检索（embedding） | 去重 + 更新 + 遗忘 + 配额限制 |

| 关键设计决策 | 做法 | 原因 |
|-------------|------|------|
| LLM 判断存什么 | 每轮对话后 LLM 做记忆提取审查 | 规则判断太僵硬，LLM 理解语义才能区分"值得记"和"噪音" |
| 不阻塞对话 | `asyncio.create_task` 后台执行 | 记忆提取耗时 1~3 秒，不能拖慢用户看到的回复 |
| 上下文预算 | 各组件按比例分配 token 份额 | 防止任一组件膨胀挤占全部空间 |
| 对话压缩 | 旧消息摘要 + 最近消息完整 | 既保留对话连贯性，又控制 token 数 |
| embedding 选型 | text-embedding-3-small | 性价比最高，$0.02/1M token，1536 维检索效果好 |

**一句话总结：短期记忆是"记住刚才说了什么"，长期记忆是"记住你是一个什么样的人"。前者靠 Redis + 裁剪压缩，后者靠 LLM 提取 + Milvus 语义存储。两者协同，Agent 才能既有连贯对话能力，又有跨会话的个性化记忆。上下文预算系统是防止两者合在一起撑爆 prompt 的最终防线。**
