# 第四章：工作流

> **核心论点**：单次 LLM 调用能解决的问题有限。真正的 Agent 能力来自于把多个 LLM 调用、工具执行、条件判断串联成一条可控的执行链——这就是工作流。本章聚焦 LangGraph 工作流的搭建方法，以及工作流中最棘手的问题：如何保证上游 LLM 节点的输出是稳定的，下游节点能可靠解析。

---

## 4.1 工作流是什么——从"一问一答"到"多步协作"

### 4.1.1 没有工作流的样子

```python
# 没有工作流：一个巨大的 prompt，一次 LLM 调用，一次返回
@app.post("/analyze")
async def analyze(text: str):
    prompt = f"""
    请分析以下文本：
    1. 提取关键信息
    2. 判断情感倾向
    3. 生成摘要
    4. 翻译成英文
    
    文本：{text}
    """
    response = await llm.ainvoke(prompt)
    return {"result": response.content}
```

问题：
- 每一步的质量无法单独控制
- 某一步失败，全部重来
- 无法在中间步骤插入工具调用
- LLM 的输出格式不稳定（有时 Markdown，有时纯文本，有时 JSON），难以解析
- Prompt 越长，LLM 的注意力越分散，质量越差

### 4.1.2 有工作流的样子

```
用户文本
   │
   ▼
┌──────────┐    结构化信息
│ 信息提取  │──────────────┐
│ (Node 1) │              │
└──────────┘              ▼
                    ┌──────────┐    情感标签
                    │ 情感分析  │──────────────┐
                    │ (Node 2) │              │
                    └──────────┘              ▼
                                        ┌──────────┐    摘要文本
                                        │ 摘要生成  │──────────────┐
                                        │ (Node 3) │              │
                                        └──────────┘              ▼
                                                            ┌──────────┐
                                                            │ 翻译     │
                                                            │ (Node 4) │
                                                            └────┬─────┘
                                                                 │
                                                                 ▼
                                                            最终结果
```

每个节点只做一件事，输入和输出都有明确的 Schema。节点之间通过**共享状态（State）**传递数据。这就是 LangGraph 的核心理念。

---

## 4.2 LangGraph 工作流基础：StateGraph 三要素

### 4.2.1 要素一：State（状态）——节点间的数据总线

State 是一个共享字典，每个节点读写它。LangGraph 会自动管理 State 在节点间的传递。

```python
from typing import TypedDict, Annotated
from operator import add
from langgraph.graph import StateGraph


# 定义 State 的结构
class AnalysisState(TypedDict):
    text: str                         # 输入文本
    entities: dict                    # 提取的结构化信息（Node 1 输出）
    sentiment: str                    # 情感标签（Node 2 输出）
    summary: str                      # 摘要（Node 3 输出）
    translation: str                  # 英文翻译（Node 4 输出）
    # 使用 Annotated + add 实现列表的追加（而非覆盖）
    messages: Annotated[list, add]    # 对话消息列表


# 初始 State
initial_state = AnalysisState(
    text="今天天气真好，适合出去走走。",
    entities={},
    sentiment="",
    summary="",
    translation="",
    messages=[],
)
```

关键设计：State 中的字段默认是**覆盖**语义（新值替换旧值）。如果需要**追加**语义（如消息列表），用 `Annotated[list, add]`。

### 4.2.2 要素二：Node（节点）——执行单元

每个节点是一个普通的 Python 函数，接收 State，返回 State 的部分更新：

```python
async def extract_entities(state: AnalysisState) -> dict:
    """
    节点 1：信息提取。
    输入 → State，输出 → State 的部分更新（只返回变更的字段）。
    """
    result = await llm.ainvoke(f"提取以下文本的关键实体...{state['text']}")
    entities = parse_entities(result.content)
    
    # 只返回这个节点负责更新的字段
    return {"entities": entities}


async def analyze_sentiment(state: AnalysisState) -> dict:
    """节点 2：情感分析。可以从 state 中读取上游节点的输出。"""
    # 用 entities 辅助情感判断
    context = f"文本：{state['text']}\n关键实体：{state['entities']}"
    result = await llm.ainvoke(f"判断情感...{context}")
    
    return {"sentiment": result.content}
```

节点的返回值是 `dict`，LangGraph 会自动将其合并回 State。只返回变更的字段即可，不需要返回整个 State。

### 4.2.3 要素三：Edge（边）——控制流的连接

```python
# 构建工作流
graph = StateGraph(AnalysisState)

# 添加节点
graph.add_node("extract", extract_entities)
graph.add_node("sentiment", analyze_sentiment)
graph.add_node("summarize", generate_summary)
graph.add_node("translate", translate_to_en)

# 添加边（定义执行顺序）
graph.add_edge("extract", "sentiment")       # extract → sentiment
graph.add_edge("sentiment", "summarize")     # sentiment → summarize
graph.add_edge("summarize", "translate")     # summarize → translate

# 设置入口和出口
graph.set_entry_point("extract")
graph.set_finish_point("translate")

# 编译
app = graph.compile()
```

边有三种类型：

| 类型 | 用法 | 含义 |
|------|------|------|
| 普通边 | `graph.add_edge("A", "B")` | A 执行完后无条件执行 B |
| 条件边 | `graph.add_conditional_edges("A", router, {"x": "B", "y": "C"})` | A 执行完后，根据 router 函数的返回值选择下一个节点 |
| 入口/出口 | `graph.set_entry_point("A")` | 工作流的起点和终点 |

## 4.3 条件边与路由：让工作流有"判断力"

### 4.3.1 最经典的条件路由：根据用户意图分流

```python
from langgraph.graph import StateGraph

class RouterState(TypedDict):
    message: str
    intent: str
    response: str


async def classify_intent_node(state: RouterState) -> dict:
    """分类节点：判断意图，写入 state"""
    intent_result = await classify_intent(state["message"])
    return {"intent": intent_result.intent}


def route_by_intent(state: RouterState) -> str:
    """
    路由函数：根据 intent 字段的值，返回下一个节点的名称。
    
    关键：这个函数不调 LLM，只做纯逻辑判断。
    返回的字符串必须是 add_conditional_edges 中映射的 key。
    """
    if state["intent"] == "order":
        return "order_handler"
    elif state["intent"] == "question":
        return "qa_handler"
    else:
        return "chat_handler"


# 构建
graph = StateGraph(RouterState)
graph.add_node("classify", classify_intent_node)
graph.add_node("order_handler", handle_order)
graph.add_node("qa_handler", handle_qa)
graph.add_node("chat_handler", handle_chat)

graph.set_entry_point("classify")
graph.add_conditional_edges(
    "classify",          # 从哪个节点出发
    route_by_intent,     # 路由函数，返回下一个节点名
    {
        "order_handler": "order_handler",
        "qa_handler": "qa_handler",
        "chat_handler": "chat_handler",
    },
)
graph.add_edge("order_handler", END)
graph.add_edge("qa_handler", END)
graph.add_edge("chat_handler", END)
```

### 4.3.2 条件路由的关键原则

**路由函数必须是纯逻辑函数，不能再调 LLM。** 因为路由函数的返回值直接决定下一个节点，如果路由函数内部又调了 LLM，那 LLM 的一次不稳定输出就会导致走错节点。判断依据应该已经在 State 里（由前面的节点写入），路由函数只做 `if/elif/else`。

```python
# ✅ 路由函数：纯逻辑，不调 LLM
def route_by_sentiment(state: AnalysisState) -> str:
    if state["sentiment"] == "negative":
        return "escalate"
    return "continue"

# ❌ 路由函数里调 LLM——不要这样做
async def route_by_llm(state: AnalysisState) -> str:
    result = await llm.ainvoke(f"这个要不要升级？{state['sentiment']}")
    return "escalate" if "升级" in result.content else "continue"
```

## 4.4 工作流的核心难题：保证 LLM 输出稳定可解析

### 4.4.1 问题场景

工作流中最常见的崩溃模式：

```
Node 1（LLM 提取）：
  输入：文本
  期望输出：{"name": "张三", "age": 30, "city": "北京"}
  实际输出："好的，我提取到以下信息：\n姓名：张三\n年龄：30\n城市：北京"
              ↑ LLM 用 Markdown 格式输出了，程序 JSON.parse 直接崩
              
Node 2（解析）：json.loads(state["entities"])  →  💥 JSONDecodeError
```

这就是工作流中最大的风险点——一个节点的输出是另一个节点的输入，如果上游输出的格式不稳定，下游就会崩溃。而 LLM 天生就是"格式不稳定"的。

### 4.4.2 方案一：`with_structured_output`（LangChain 原生方案，推荐）

```python
from pydantic import BaseModel, Field
from langchain_openai import ChatOpenAI


class ExtractedEntities(BaseModel):
    """Node 1 的期望输出结构"""
    name: str = Field(default="", description="人物姓名")
    age: int = Field(default=0, description="年龄")
    city: str = Field(default="", description="所在城市")
    keywords: list[str] = Field(default_factory=list, description="关键词列表")


# 使用 with_structured_output 绑定输出结构
llm = ChatOpenAI(model="gpt-4o")
structured_llm = llm.with_structured_output(ExtractedEntities)


async def extract_entities_node(state: AnalysisState) -> dict:
    """节点：信息提取。输出保证是 ExtractedEntities 类型。"""
    result = await structured_llm.ainvoke(
        f"从以下文本中提取人物信息：\n{state['text']}"
    )
    # result 是 ExtractedEntities 实例，不是字符串
    # 不需要 json.loads，不需要 try-catch 解析
    return {"entities": result.model_dump()}
```

`with_structured_output` 做了什么：
1. 在 API 请求中使用 `response_format` 参数（OpenAI 的 Structured Outputs 或 JSON Mode）
2. LLM 返回后，自动用 Pydantic 校验
3. 如果 LLM 输出的 JSON 不合法，自动重试（内部机制）
4. 返回给你的就是 Pydantic 实例，类型安全

**这是保证节点间数据传递稳定性的第一选择。** 能用 `with_structured_output` 的地方优先用它，不要手写 JSON 解析。

### 4.4.3 方案二：手动 JSON Mode + Pydantic 二次校验

如果用的模型不支持 `with_structured_output`，或者你需要更灵活的控制：

```python
import json
from pydantic import ValidationError
from core.llm import call_llm_with_retry


async def extract_entities_node(state: AnalysisState) -> dict:
    """手动 JSON 输出 + Pydantic 校验 + 重试"""
    
    prompt = f"""从以下文本中提取人物信息，以 JSON 格式返回。

文本：{state['text']}

输出格式：
{{
    "name": "姓名（没有则为空字符串）",
    "age": 年龄数字（没有则为0）,
    "city": "城市（没有则为空字符串）",
    "keywords": ["关键词1", "关键词2"]
}}

只输出 JSON，不要添加任何解释文字。"""

    for attempt in range(3):
        try:
            response = await call_llm_with_retry(
                messages=[{"role": "user", "content": prompt}],
                response_format={"type": "json_object"},  # 强制 JSON 输出
            )
            
            # 第一层：JSON 解析
            raw = json.loads(response)
            
            # 第二层：Pydantic 校验
            entities = ExtractedEntities(**raw)
            
            return {"entities": entities.model_dump()}
            
        except json.JSONDecodeError as e:
            logger.warning("Node1 JSON解析失败 (attempt=%d): %s", attempt + 1, str(e))
            if attempt == 2:
                # 三次都失败，返回空结构，不阻塞整个工作流
                return {"entities": ExtractedEntities().model_dump()}
            
        except ValidationError as e:
            logger.warning("Node1 Pydantic校验失败 (attempt=%d): %s", attempt + 1, str(e))
            if attempt == 2:
                return {"entities": ExtractedEntities().model_dump()}
```

**双重校验是保险机制：** JSON 解析保证是合法 JSON，Pydantic 校验保证字段类型正确。两关都通过，下游节点才能放心使用。

### 4.4.4 方案三：在 prompt 里用 Few-shot 示例固化输出格式

对于不支持 JSON Mode 的模型，或者输出特别复杂的情况，Few-shot 是最可靠的手段：

```python
EXTRACTION_PROMPT = """从文本中提取人物信息。

## 输出格式（严格遵守）
<entity>
name: 姓名
age: 年龄
city: 城市
</entity>

## 示例
文本：我叫李四，今年25岁，住在上海。
<entity>
name: 李四
age: 25
city: 上海
</entity>

文本：小红是杭州人。
<entity>
name: 小红
age: 0
city: 杭州
</entity>

## 现在请提取
文本：{text}"""
```

然后解析时用正则或简单字符串匹配：

```python
import re

def parse_entity_output(raw: str) -> dict:
    """解析标记格式的输出，比 JSON 更宽松"""
    result = {"name": "", "age": 0, "city": ""}
    
    for line in raw.split("\n"):
        line = line.strip()
        if line.startswith("name:"):
            result["name"] = line.split(":", 1)[1].strip()
        elif line.startswith("age:"):
            try:
                result["age"] = int(line.split(":", 1)[1].strip())
            except ValueError:
                pass
        elif line.startswith("city:"):
            result["city"] = line.split(":", 1)[1].strip()
    
    return result
```

这种方式比 JSON 容错性好——LLM 多了一个空格、少了一个引号都不会崩。但只适合简单结构，复杂嵌套结构还是推荐 JSON。

### 4.4.5 三个方案的选型指南

| 方案 | 可靠性 | 复杂度 | 适用场景 |
|------|--------|--------|---------|
| `with_structured_output` | ★★★★★ | 最低 | GPT-4o/Claude 等强模型，首选 |
| JSON Mode + Pydantic 双重校验 | ★★★★☆ | 中等 | 需要灵活控制 prompt，或模型不支持 structured output |
| Few-shot 标记格式 | ★★★☆☆ | 低（解析简单，但 LLM 输出可能偏离格式） | 简单结构、弱模型、需要最大容错 |

**原则：能用 `with_structured_output` 就优先用它。一个 API 调用就能解决的事，不要自己写解析和重试。**

## 4.5 节点间的数据契约：下游节点必须"防御性读取"

即使上游用了 `with_structured_output`，下游节点也不能假设数据一定完美：

```python
async def sentiment_analysis_node(state: AnalysisState) -> dict:
    """
    下游节点：防御性读取上游数据。
    
    即使上游合约是 ExtractedEntities，也要处理：
    - 实体为空
    - 字段缺失
    - 类型不对（上游出了 bug）
    """
    entities = state.get("entities", {})
    
    # 防御性读取：给默认值，做类型断言
    name = entities.get("name", "")
    age = entities.get("age", 0)
    city = entities.get("city", "")
    
    # 确保类型正确
    if not isinstance(age, int):
        age = 0
    
    # 构建分析上下文（无实体也能正常工作）
    context_parts = []
    if name:
        context_parts.append(f"人物：{name}")
    if city:
        context_parts.append(f"城市：{city}")
    
    context = "；".join(context_parts) if context_parts else "无额外上下文"
    
    result = await llm.ainvoke(
        f"文本：{state['text']}\n{context}\n判断上述文本的情感倾向..."
    )
    
    return {"sentiment": result.content.strip()}
```

**规则：上游节点的输出对下游节点来说是不可信的。** 即使类型系统保证了结构，也要做最低限度的防御——给默认值、校验类型、处理空值。这是工作流稳定性的最后一道防线。

## 4.6 实战：搭建一个完整的三节点工作流

### 4.6.1 场景

用户提交一段客服对话记录，Agent 需要：
1. 提取客户信息 + 问题描述
2. 判断问题严重程度
3. 严重问题 → 生成升级处理建议，普通问题 → 生成标准回复模板

### 4.6.2 完整代码

```python
# workflows/ticket_analysis.py
from typing import TypedDict, Literal
from pydantic import BaseModel, Field
from langgraph.graph import StateGraph, END
from langchain_openai import ChatOpenAI
from core.logger import get_logger

logger = get_logger(__name__)
llm = ChatOpenAI(model="gpt-4o")

# ==================== State 定义 ====================

class TicketState(TypedDict):
    transcript: str                   # 原始对话记录
    customer_name: str                # 客户姓名
    customer_level: str               # 客户等级
    problem: str                      # 问题描述
    severity: str                     # 严重程度：critical / normal
    action: str                       # 最终输出


# ==================== 结构化输出模型 ====================

class ExtractionResult(BaseModel):
    """Node 1 的输出结构"""
    customer_name: str = Field(default="", description="客户姓名，没有则为空")
    customer_level: str = Field(default="普通", description="客户等级：VIP/普通")
    problem: str = Field(description="问题的简要描述（一句话）")


class SeverityResult(BaseModel):
    """Node 2 的输出结构"""
    severity: Literal["critical", "normal"] = Field(description="严重程度")
    reason: str = Field(description="判断依据")


# ==================== Node 1：信息提取 ====================

extract_llm = llm.with_structured_output(ExtractionResult)

async def extract_info(state: TicketState) -> dict:
    """从对话记录中提取客户和问题信息"""
    try:
        result = await extract_llm.ainvoke(
            f"分析以下客服对话记录，提取客户信息和问题描述：\n\n{state['transcript']}"
        )
        logger.info("信息提取完成 | customer=%s | level=%s", result.customer_name, result.customer_level)
        return result.model_dump()
    except Exception:
        logger.exception("信息提取失败")
        # 兜底：返回空结果，工作流继续
        return {"customer_name": "", "customer_level": "普通", "problem": "信息提取失败"}


# ==================== Node 2：严重程度判断 ====================

severity_llm = llm.with_structured_output(SeverityResult)

async def assess_severity(state: TicketState) -> dict:
    """根据客户等级和问题描述，判断严重程度"""
    # 防御性读取
    level = state.get("customer_level", "普通")
    problem = state.get("problem", "")
    
    try:
        result = await severity_llm.ainvoke(
            f"客户等级：{level}\n问题：{problem}\n\n判断该问题的严重程度。"
            f"判定标准：VIP客户 = critical；涉及退款/投诉 = critical；其他 = normal。"
        )
        logger.info("严重程度判定 | severity=%s | reason=%s", result.severity, result.reason)
        return {"severity": result.severity}
    except Exception:
        logger.exception("严重程度判定失败")
        return {"severity": "normal"}  # 默认普通


# ==================== Node 3A：严重问题处理 ====================

async def handle_critical(state: TicketState) -> dict:
    """生成升级处理建议"""
    response = await llm.ainvoke(
        f"生成客户投诉升级处理方案：\n"
        f"客户：{state.get('customer_name', '未知')}（{state.get('customer_level', '普通')}）\n"
        f"问题：{state.get('problem', '未知')}\n"
        f"要求：1. 升级到主管 2. 48小时内闭环 3. 提供补偿方案建议"
    )
    return {"action": response.content}


# ==================== Node 3B：普通问题处理 ====================

async def handle_normal(state: TicketState) -> dict:
    """生成标准回复模板"""
    response = await llm.ainvoke(
        f"生成客服标准回复模板：\n"
        f"客户：{state.get('customer_name', '用户')}\n"
        f"问题：{state.get('problem', '未知')}\n"
        f"要求：礼貌、专业、给出预计解决时间"
    )
    return {"action": response.content}


# ==================== 路由函数 ====================

def route_by_severity(state: TicketState) -> str:
    """纯逻辑路由：不调 LLM"""
    if state.get("severity") == "critical":
        return "critical_handler"
    return "normal_handler"


# ==================== 构建工作流 ====================

def build_ticket_workflow():
    graph = StateGraph(TicketState)

    # 添加节点
    graph.add_node("extract", extract_info)
    graph.add_node("assess", assess_severity)
    graph.add_node("critical_handler", handle_critical)
    graph.add_node("normal_handler", handle_normal)

    # 连线
    graph.set_entry_point("extract")
    graph.add_edge("extract", "assess")
    graph.add_conditional_edges(
        "assess",
        route_by_severity,
        {"critical_handler": "critical_handler", "normal_handler": "normal_handler"},
    )
    graph.add_edge("critical_handler", END)
    graph.add_edge("normal_handler", END)

    return graph.compile()


ticket_workflow = build_ticket_workflow()
```

调用方式：

```python
result = await ticket_workflow.ainvoke({
    "transcript": "客户张三（VIP）：我买的手机屏幕碎了，要求退款！客服：好的我记录一下...",
    "customer_name": "",
    "customer_level": "",
    "problem": "",
    "severity": "",
    "action": "",
})
# → result["severity"] = "critical"
# → result["action"] = "升级处理方案：赔偿 + 主管跟进..."
```

### 4.6.3 这个工作流的"稳定合约"体现在哪

```
Node 1 (extract_info)
  输出: ExtractionResult (Pydantic 校验保证)
  异常: 返回默认空结构，不崩
         │
         ▼
Node 2 (assess_severity)
  输入: state.customer_name / state.problem
  读取方式: state.get("problem", "")  ← 防御性，给默认值
  输出: SeverityResult (Pydantic 校验保证)
  异常: 返回 severity="normal"
         │
         ▼
路由函数 (route_by_severity)
  输入: state.severity  ← 一定是 "critical" 或 "normal"
  逻辑: if/else 纯判断
         │
    ┌────┴────┐
    ▼         ▼
Node 3A     Node 3B
  输入: state 各字段（全有默认值保护）
```

每一步都有两重保障：**Pydantic 校验保证输出结构** + **`.get(key, default)` 保证输入容错**。任何单节点的崩溃都不会导致整个工作流中断。

## 4.7 工作流的三种常见模式

### 4.7.1 模式一：串行管道（Pipeline）

```
A → B → C → D → 结束
```

适用场景：数据需要逐步加工（提取 → 分析 → 总结 → 翻译）。前面已有完整示例。

### 4.7.2 模式二：并行扇出/聚合（Fan-out / Fan-in）

```
              ┌→ B1（查订单）─┐
A（意图识别）→ ┼→ B2（查库存）─┼→ C（汇总回复）
              └→ B3（查物流）─┘
```

LangGraph 原生支持并行：

```python
# B1, B2, B3 会同时执行
graph.add_edge("A", "B1")
graph.add_edge("A", "B2")
graph.add_edge("A", "B3")

# 都执行完后汇入 C
graph.add_edge("B1", "C")
graph.add_edge("B2", "C")
graph.add_edge("B3", "C")
```

从同一个节点出发的多条边，LangGraph 会自动并行执行目标节点。只有所有并行节点都完成后，才会进入汇聚节点 C。

### 4.7.3 模式三：ReAct 循环（思考-行动-观察）

```
     ┌──────────────────────────┐
     │                          │
     ▼                          │
┌─────────┐   tool_call   ┌─────────┐
│  LLM    │──────────────→│ 工具执行 │
│  推理   │←──────────────│         │
└─────────┘   结果文本    └─────────┘
     │
     │ 无 tool_call
     ▼
   结束
```

这是 Agent 最经典的模式，LangGraph 的 `create_react_agent` 一行代码就能创建：

```python
from langgraph.prebuilt import create_react_agent

agent = create_react_agent(
    model=llm,
    tools=[order_tool, inventory_tool, search_kb],
)

result = await agent.ainvoke({"messages": [{"role": "user", "content": "查一下订单 ORD-001"}]})
```

对于自定义的 ReAct 循环，需要用到条件边：

```python
def should_continue(state: AgentState) -> str:
    """检查最后一条消息是否有 tool_calls"""
    last_message = state["messages"][-1]
    if hasattr(last_message, "tool_calls") and last_message.tool_calls:
        return "tools"
    return "end"

graph.add_conditional_edges(
    "llm",
    should_continue,
    {"tools": "tools", "end": END},
)
graph.add_edge("tools", "llm")  # 工具执行完 → 回到 LLM 继续推理
```

## 4.8 工作流调试与可观测性

### 4.8.1 记录每个节点的输入输出

```python
from functools import wraps
from core.logger import get_logger

logger = get_logger("workflow")


def trace_node(node_name: str):
    """节点追踪装饰器：自动记录输入输出的关键信息"""
    def decorator(func):
        @wraps(func)
        async def wrapper(state: dict) -> dict:
            logger.info("[%s] 开始执行", node_name)
            
            try:
                result = await func(state)
                
                # 只记录输出字段名和长度，不记录完整内容（避免日志爆炸）
                output_summary = {
                    k: f"len={len(str(v))}" if isinstance(v, (str, list, dict)) else str(v)
                    for k, v in result.items()
                }
                logger.info("[%s] 执行完成 | output=%s", node_name, output_summary)
                
                return result
            except Exception:
                logger.exception("[%s] 执行异常", node_name)
                raise
        
        return wrapper
    return decorator


# 使用
@trace_node("extract_info")
async def extract_info(state: TicketState) -> dict:
    ...
```

### 4.8.2 用 LangGraph 的 `astream_events` 观测整个流程

```python
async def run_with_trace(workflow, initial_state: dict):
    """运行工作流并流式输出每个节点的事件"""
    async for event in workflow.astream_events(initial_state, version="v2"):
        kind = event["event"]
        
        if kind == "on_chain_start":
            name = event.get("name", "unknown")
            logger.info("▶ 节点开始: %s", name)
        
        elif kind == "on_chain_end":
            name = event.get("name", "unknown")
            output = event.get("data", {}).get("output", {})
            logger.info("■ 节点完成: %s", name)
        
        elif kind == "on_chain_error":
            name = event.get("name", "unknown")
            error = event.get("data", {}).get("error", "")
            logger.error("✖ 节点异常: %s | %s", name, error)
```

## 4.9 常见踩坑清单

| 坑 | 现象 | 原因 | 解法 |
|----|------|------|------|
| LLM 输出格式不对 | 下游节点 json.loads 报错 | LLM 在 JSON 前后加了 Markdown 标记或解释文字 | 用 `with_structured_output`，或在 prompt 中强调"只输出 JSON"，并做 Pydantic 二次校验 |
| 并行节点数据竞争 | 并行执行的两个节点都在改 state 的同一个字段 | 两个节点同时写同一个 key，后者覆盖前者 | 并行节点写不同的字段，或使用 `Annotated[list, add]` 追加语义 |
| 条件路由走错 | 该升级的工单没升级 | 路由函数基于 LLM 输出判断，LLM 输出不稳定 | 路由函数只做纯逻辑判断，判断依据必须提前写入 State |
| 节点异常导致整个工作流崩溃 | 一个节点抛异常，工作流终止 | 节点没有 try-catch，LangGraph 默认向上传播异常 | 每个节点内部 try-catch，异常时返回默认值，不阻塞流程 |
| State 字段拼写错误 | 下游节点读不到上游写的数据 | `return {"severity": "high"}` 但下游读 `state["severty"]` | 用 TypedDict 定义 State，IDE 自动补全 + mypy 静态检查 |
| 工作流无限循环 | ReAct 循环跑了 100 步不结束 | LLM 反复调同一个工具 | `recursion_limit` + `ToolCallGuard`（见第三章） |

## 4.10 本章小结

| 要点 | 核心做法 | 一句话 |
|------|---------|--------|
| State 设计 | TypedDict 定义所有字段，明确每个字段由哪个节点写入 | State 是节点间的数据总线，结构清晰是第一要务 |
| 结构化输出 | `llm.with_structured_output(PydanticModel)` | 让 LLM 直接输出 Pydantic 实例，省掉解析和校验代码 |
| 数据契约 | 上游 Pydantic 校验输出 + 下游 `.get(key, default)` 防御读取 | 上游保证输出正确，下游保证输入容错——双重保险 |
| 条件路由 | 路由函数做纯逻辑判断，不调 LLM | 路由依据必须在 State 中已存在，路由函数只做 if/else |
| 节点异常 | 每个节点内部 try-catch，异常返回默认值 | 一个节点崩了不能拖整个工作流陪葬 |
| 并行执行 | 从同一节点出发的多条边自动并行 | 并行节点之间不能写 State 的同一个字段 |

**工作流的本质是"把复杂任务拆成多个简单步骤，每一步只做一件事，但要保证上一步的输出能安全地成为下一步的输入"。** 拆得开是设计能力，接得稳是工程质量。
