# 第三章：工具集

> **核心论点**：工具是 Agent 的手和脚。没有工具，模型只能"说说而已"；有了工具，模型能查数据库、调 API、发邮件、操作文件。但工具调用也是最容易失控的环节——模型可能反复重试一个注定失败的工具、可能误解用户意图乱调工具、可能陷入无限循环。本章逐一拆解工具的注册、调用、意图识别和失控防范。

---

## 3.1 LangChain 的工具定义：三种方式与选择

### 3.1.1 方式一：`@tool` 装饰器（推荐，最常用）

```python
from langchain_core.tools import tool
from pydantic import BaseModel, Field


# 最简单的用法：装饰一个 async 函数
@tool
async def get_weather(city: str) -> str:
    """查询城市的实时天气。city: 城市名称，例如'北京'或'Shanghai'。"""
    # 实际调天气 API...
    return f"{city}：晴，22°C，湿度 45%"


# 带 Pydantic 参数校验的用法
class WeatherInput(BaseModel):
    city: str = Field(description="城市名称，支持中文或英文")
    date: str = Field(default="today", description="日期，格式 YYYY-MM-DD，默认今天")


@tool(args_schema=WeatherInput)
async def get_weather_v2(city: str, date: str = "today") -> str:
    """查询指定城市指定日期的天气。"""
    return f"{city} {date}：晴，22°C"
```

`@tool` 装饰器做了什么：
1. 读取函数名作为工具名（`get_weather`）
2. 读取 docstring 作为工具描述（发给 LLM 看）
3. 读取函数参数签名 + 类型注解，自动生成 JSON Schema
4. 包装为 `BaseTool` 子类实例

### 3.1.2 方式二：继承 `BaseTool`（需要更多控制时用）

```python
from langchain_core.tools import BaseTool
from pydantic import BaseModel, Field
from typing import Type


class OrderSearchInput(BaseModel):
    order_id: str = Field(description="订单号，格式 ORD-XXXXX")


class OrderSearchTool(BaseTool):
    name: str = "search_order"
    description: str = "根据订单号查询订单详情，返回订单状态、金额和物流信息。"
    args_schema: Type[BaseModel] = OrderSearchInput

    async def _arun(self, order_id: str) -> str:
        """异步执行（Agent 调用入口）"""
        return await self._search(order_id)

    def _run(self, order_id: str) -> str:
        """同步执行（一般用不到，Agent 调用的是 _arun）"""
        return asyncio.run(self._search(order_id))

    async def _search(self, order_id: str) -> str:
        # 实际查询逻辑
        ...
```

**什么时候用继承而不是装饰器：**
- 需要工具内部维护状态（如连接池、计数器）
- 需要 `name` 和函数名不同
- 需要在 `_arun` 前后做统一的预处理/后处理
- 需要动态决定是否返回异常或结果

### 3.1.3 方式三：`StructuredTool`（从普通函数快速构造）

```python
from langchain_core.tools import StructuredTool


async def search_docs(query: str, source: str = "all") -> str:
    """搜索文档"""
    ...


# 如果已有普通函数，用 StructuredTool 包装
doc_tool = StructuredTool.from_function(
    coroutine=search_docs,
    name="search_documents",
    description="在内部文档库中搜索。query: 关键词；source: 来源，可选 all/wiki/faq",
)
```

**三种方式的选择：**

| 方式 | 适用场景 | 代码量 |
|------|---------|--------|
| `@tool` 装饰器 | 90% 的工具 | 最少 |
| 继承 `BaseTool` | 需要内部状态、复杂初始化 | 最多 |
| `StructuredTool` | 已有函数不想改签名、动态创建工具 | 中等 |

## 3.2 工具如何绑定到模型——`bind_tools` 的机制

### 3.2.1 核心流程

```
定义工具 → bind_tools 绑定到 ChatModel → Agent 调用时 LLM 决定是否调工具
                                              │
                                    返回 tool_calls ↓
                                              │
                                    执行工具 → 结果返给 LLM → 继续推理
```

```python
from langchain_openai import ChatOpenAI
from langchain_core.tools import tool

# 1. 定义工具
@tool
async def calculator(expression: str) -> str:
    """执行数学计算。expression: 数学表达式，如 '2+3*4'。"""
    ...

@tool
async def search_web(query: str) -> str:
    """搜索互联网。query: 搜索关键词。"""
    ...

tools = [calculator, search_web]

# 2. 绑定到模型
llm = ChatOpenAI(model="gpt-4o")
llm_with_tools = llm.bind_tools(tools)

# 3. 调用 —— LLM 会自动判断是否需要工具
response = await llm_with_tools.ainvoke([
    {"role": "user", "content": "计算 123 * 456 等于多少"}
])
# → response.tool_calls = [{"name": "calculator", "args": {"expression": "123*456"}}]
```

### 3.2.2 `bind_tools` 实际做了什么

`bind_tools` 并不是把工具代码"塞给"模型。它做的是：

1. 把每个工具的 `name`、`description`、`args_schema`（参数 JSON Schema）转为 OpenAI function calling 格式
2. 在每次 API 请求时把这些工具定义作为 `tools` 参数发送
3. LLM 根据用户消息和工具描述，决定是否返回 `tool_calls`

```
发送给 OpenAI 的请求体（简化）：
{
  "model": "gpt-4o",
  "messages": [{"role": "user", "content": "计算 123 * 456"}],
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "calculator",
        "description": "执行数学计算。expression: 数学表达式...",
        "parameters": {
          "type": "object",
          "properties": {
            "expression": {"type": "string", "description": "数学表达式"}
          },
          "required": ["expression"]
        }
      }
    },
    // ... 其他工具
  ]
}
```

**关键认知：工具定义是跟着每一次 LLM 请求发送的。** 10 个工具，每次请求都在消耗约 500~2000 token 的工具定义开销。这就是为什么工具描述要简洁。

### 3.2.3 `tool_choice` —— 控制模型是否必须调工具

```python
# auto（默认）：LLM 自己判断要不要调工具
llm.bind_tools(tools, tool_choice="auto")

# required：强制 LLM 必须调工具（不调就报错）
llm.bind_tools(tools, tool_choice="required")

# none：禁止 LLM 调工具（纯文本回复）
llm.bind_tools(tools, tool_choice="none")

# 指定工具：强制 LLM 必须调某个具体工具
llm.bind_tools(tools, tool_choice={"type": "function", "function": {"name": "calculator"}})
```

**使用建议：**
- C 端对话 Agent：用 `"auto"`，让 LLM 自然判断
- 特定任务节点（如"查询订单"节点）：用 `"required"` 或指定工具名
- 纯文本回复阶段（Agent 已拿到工具结果，需要整合输出）：用 `"none"`
- 路由/分发节点：可以指定工具，但更推荐用意图识别（见 3.6 节）

## 3.3 工具调用的准确性问题

### 3.3.1 第一类问题：参数填错

LLM 可能传了不符合工具预期的参数：

```python
# LLM 产生了这样的 tool_call：
{"name": "get_weather", "args": {"city": "北京", "date": "明天"}}
#                                                         └─ 工具要求 "YYYY-MM-DD"

# ❌ 工具直接崩溃
@tool
async def get_weather(city: str, date: str) -> str:
    datetime.strptime(date, "%Y-%m-%d")  # ValueError!
    ...

# ✅ 工具做参数容错
@tool
async def get_weather(city: str, date: str = "today") -> str:
    try:
        if date == "today" or date == "今天":
            date = datetime.now().strftime("%Y-%m-%d")
        elif date == "tomorrow" or date == "明天":
            date = (datetime.now() + timedelta(days=1)).strftime("%Y-%m-%d")
        else:
            datetime.strptime(date, "%Y-%m-%d")  # 严格校验
    except ValueError:
        return f"日期格式错误：'{date}'。请使用 YYYY-MM-DD 格式。"  # 返回文本，不抛异常
    ...
```

### 3.3.2 第二类问题：工具执行失败后模型反复重试（最重要）

这是 Agent 工具调用最常见的生产事故：

```
用户："帮我查订单 ORD-12345"
  → LLM 调用 search_order("ORD-12345")
  → 数据库连接池满了，工具抛异常
  → LLM 看到异常，再次调用 search_order("ORD-12345")
  → 再次失败
  → LLM 再次调用...
  → 循环 10 次，每次都是同样的失败，Token 全浪费
```

**根本原因：LLM 不知道"这个失败是不可恢复的"。** 它看到一个错误，本能反应是"我再试一次"，就像人类看到网页加载失败会刷新一样。

解法分三层：

**第一层：工具绝不抛异常，返回描述性失败文本（第二章 1.8 节已讲，这里强化）**

```python
@tool
async def search_order(order_id: str) -> str:
    try:
        result = await db_query(...)
        return format_order(result)
    except ConnectionRefusedError:
        # 关键：明确告诉 LLM"不要重试"
        return (
            "[工具执行失败]\n"
            f"订单查询失败，原因：数据库连接被拒绝。\n"
            f"此错误无法通过重试解决，请勿再次调用此工具。\n"
            f"请直接告知用户'订单系统暂时不可用，请稍后再试'。"
        )
    except RecordNotFoundError:
        return f"未找到订单 {order_id}。请确认订单号是否正确，不要重复查询。"
```

**第二层：Agent 层面设置 `recursion_limit`**

```python
from langgraph.graph import StateGraph

graph = StateGraph(...)
# ...
compiled = graph.compile()

# 调用时限制最大步数
result = await compiled.ainvoke(
    {"messages": [...]},
    config={"recursion_limit": 15},  # 最多 15 步（LLM推理+工具调用合计）
)
```

`recursion_limit` 不是专门针对工具调用的，但可以防止无限循环。当 Agent 步数超限，LangGraph 会抛出 `GraphRecursionError`，被路由层的 try-catch 兜住。

**第三层：检测重复调用，主动拦截**

```python
# core/tool_guard.py
from collections import defaultdict
from core.logger import get_logger

logger = get_logger(__name__)


class ToolCallGuard:
    """
    工具调用守卫：检测 Agent 是否在重复调用同一个工具。

    如果同一个工具 + 同一参数被连续调用超过 N 次，说明陷入死循环，
    直接拦截并返回指令让 LLM 停止。
    """

    def __init__(self, max_identical_calls: int = 2):
        self.max_identical = max_identical_calls
        self._call_history: dict[str, list[dict]] = defaultdict(list)

    def record_and_check(self, thread_id: str, tool_name: str, tool_args: dict) -> str | None:
        """
        记录一次工具调用，检查是否重复。

        返回 None 表示允许执行；
        返回字符串表示已被拦截，该字符串应直接作为工具返回值给 LLM。
        """
        key = f"{thread_id}:{tool_name}"
        args_signature = str(sorted(tool_args.items()))

        history = self._call_history[key]

        # 统计连续相同参数的调用次数
        consecutive = 0
        for call in reversed(history):
            if call["args"] == args_signature:
                consecutive += 1
            else:
                break

        if consecutive >= self.max_identical:
            logger.warning(
                "工具重复调用被拦截 | thread=%s | tool=%s | args=%s | count=%d",
                thread_id, tool_name, args_signature, consecutive + 1,
            )
            return (
                f"[工具调用被拦截]\n"
                f"工具 '{tool_name}' 已用相同参数连续调用 {consecutive + 1} 次，均以失败告终。\n"
                f"请停止调用此工具，基于已有信息回复用户，或告知用户当前操作无法完成。"
            )

        # 记录本次调用
        history.append({"args": args_signature})
        # 限制历史长度，避免内存泄漏
        if len(history) > 20:
            self._call_history[key] = history[-10:]

        return None  # 放行


# 全局单例
tool_guard = ToolCallGuard(max_identical_calls=2)
```

将其集成到工具调用链路中：

```python
# core/tool_executor.py
from core.tool_guard import tool_guard


async def execute_tool_with_guard(
    thread_id: str,
    tool_name: str,
    tool_args: dict,
    tools_map: dict[str, callable],
) -> str:
    """
    带守卫的工具执行器。
    
    先检查是否重复调用，再执行，最后兜底异常。
    """
    # 第一层：重复调用守卫
    block_msg = tool_guard.record_and_check(thread_id, tool_name, tool_args)
    if block_msg is not None:
        return block_msg

    # 第二层：执行工具
    tool_func = tools_map.get(tool_name)
    if tool_func is None:
        return f"工具 '{tool_name}' 不存在。可用工具：{list(tools_map.keys())}。请选择可用工具。"

    try:
        result = await tool_func.ainvoke(tool_args)
        return str(result)

    except Exception as e:
        logger.exception("工具执行异常 | tool=%s | args=%s", tool_name, tool_args)
        return (
            f"工具 '{tool_name}' 执行异常：{type(e).__name__}。\n"
            f"请勿重试，告知用户此操作暂时无法完成。"
        )
```

### 3.3.3 第三类问题：工具返回的信息太多，淹没了 LLM 的注意力

```python
# ❌ 工具返回了 5000 字的知识库搜索结果
@tool
async def search_knowledge_base(query: str) -> str:
    results = await vector_search(query, top_k=20)
    return "\n".join(r["content"] for r in results)  # 可能 10000+ 字符
```

LLM 面对一堵文本墙，很容易忽略关键信息，或者直接"忘记"用户最初问的是什么。

**解法：工具自己做摘要、截断，只返回最相关部分。**

```python
@tool
async def search_knowledge_base(query: str, top_k: int = 5) -> str:
    """搜索知识库。query: 关键词；top_k: 返回条数，最多 5。"""
    results = await vector_search(query, top_k=min(top_k, 5))
    
    if not results:
        return "未找到相关内容。"
    
    formatted = []
    for i, r in enumerate(results, 1):
        # 每条结果截断到 300 字符
        content = r["content"][:300]
        if len(r["content"]) > 300:
            content += "..."
        formatted.append(f"{i}. [{r.get('title', '无标题')}] {content}")
    
    return "\n".join(formatted)
```

**经验值：单个工具返回值控制在 1500 字符以内。** 如果确实需要返回大量数据（如报表），让工具写入 Redis 并返回一个引用 key，让后续代码从 Redis 读取，而不是让 LLM 处理原始数据。

## 3.4 为什么推荐 `**kwargs` 写通用工具

### 3.4.1 问题场景：工具爆炸

假设你的 Agent 需要对接 3 个外部系统：订单系统、库存系统、物流系统。每个系统有 5 个操作（查询、创建、更新、删除、列表）。用传统方式要写 15 个工具：

```python
@tool  async def search_order(...): ...
@tool  async def create_order(...): ...
@tool  async def update_order(...): ...
@tool  async def delete_order(...): ...
@tool  async def list_orders(...): ...
@tool  async def search_inventory(...): ...
@tool  async def create_inventory(...): ...
# ... 15 个工具
```

问题：
- 每次 LLM 请求都要携带 15 个工具定义（≈ 2000 token）
- 工具越多，LLM 选错的概率越大
- 每加一个新系统，要新写 5 个工具
- 工具描述维护成本高

### 3.4.2 方案：一个系统一个通用工具，用 `action` 参数分发

```python
from typing import Literal, Optional
from pydantic import BaseModel, Field
from langchain_core.tools import tool


# ============ 订单系统的通用工具 ============

class OrderToolInput(BaseModel):
    action: Literal["search", "create", "update", "cancel", "list"] = Field(
        description="操作类型：search=查询单个订单, create=创建订单, update=更新订单, cancel=取消订单, list=列出订单"
    )
    order_id: Optional[str] = Field(default=None, description="订单号，search/update/cancel 时必填")
    keyword: Optional[str] = Field(default=None, description="搜索关键词，list 时可选")
    data: Optional[dict] = Field(default=None, description="订单数据，create/update 时必填")


@tool(args_schema=OrderToolInput)
async def order_tool(
    action: str,
    order_id: Optional[str] = None,
    keyword: Optional[str] = None,
    data: Optional[dict] = None,
) -> str:
    """
    订单管理工具。支持查询、创建、更新、取消和列出订单。
    
    使用场景：
    - 用户询问某个订单的状态 → action='search'，传入 order_id
    - 用户要下订单 → action='create'，传入 data
    - 用户要取消订单 → action='cancel'，传入 order_id
    - 用户要查看所有订单 → action='list'
    """
    try:
        if action == "search":
            return await _search_order(order_id)
        elif action == "create":
            return await _create_order(data)
        elif action == "update":
            return await _update_order(order_id, data)
        elif action == "cancel":
            return await _cancel_order(order_id)
        elif action == "list":
            return await _list_orders(keyword)
        else:
            return f"未知操作 '{action}'。支持的操作：search, create, update, cancel, list"
    except Exception as e:
        return f"订单操作失败（{action}）：{e}。请勿重试。"


# 同样模式：库存系统
class InventoryToolInput(BaseModel):
    action: Literal["check", "adjust", "list"] = Field(
        description="操作类型：check=查询库存, adjust=调整库存, list=列出库存"
    )
    product_id: Optional[str] = Field(default=None, description="商品 ID，check/adjust 时必填")
    quantity: Optional[int] = Field(default=None, description="调整数量，adjust 时必填")


@tool(args_schema=InventoryToolInput)
async def inventory_tool(action: str, product_id: Optional[str] = None, quantity: Optional[int] = None) -> str:
    """库存管理工具。支持查询库存、调整库存和列出全部库存。"""
    ...


# 同样模式：物流系统
class LogisticsToolInput(BaseModel):
    action: Literal["track", "estimate"] = Field(
        description="操作类型：track=追踪物流, estimate=估算送达时间"
    )
    order_id: Optional[str] = Field(default=None, description="订单号，track/estimate 时必填")


@tool(args_schema=LogisticsToolInput)
async def logistics_tool(action: str, order_id: Optional[str] = None) -> str:
    """物流查询工具。支持追踪包裹和估算送达时间。"""
    ...
```

对比效果：

| 维度 | 15 个独立工具 | 3 个通用工具 + action 分发 |
|------|-------------|--------------------------|
| 工具定义 token 开销 | ≈ 2000 | ≈ 600 |
| 新增操作 | 写新工具 + 注册 | 在 action 枚举里加一项 |
| LLM 选择准确率 | 15 选 1 → 容易错 | 3 选 1 → 准确率高 |
| 工具描述维护 | 15 个 docstring | 3 个 docstring |

### 3.4.3 `action` 枚举的设计要点

**要点一：`Literal` 类型让参数更精确**

```python
# ✅ Literal 告诉 LLM 只能选这些值
action: Literal["search", "create", "update", "cancel", "list"]

# ❌ 普通 str，LLM 可能传 "find" 或 "query" 导致匹配不到
action: str
```

**要点二：description 里写清楚使用场景**

LLM 是靠工具描述来判断"什么时候该调哪个工具"的。`action` 的每个值对应什么场景，必须在描述里写明白：

```python
action: Literal["search", "create", "cancel"] = Field(
    description="操作类型：search=查询单个订单（用户问某订单状态时）, "
                "create=创建新订单（用户要下单时）, "
                "cancel=取消订单（用户要取消时）"
)
```

**要点三：可选参数通过 `Optional` + `default=None` 区分必填和选填**

```python
order_id: Optional[str] = Field(default=None, description="订单号，search/cancel 时必填")
keyword: Optional[str] = Field(default=None, description="搜索关键词，list 时可选")
```

LLM 看到 `required` 属性会自动判断哪些参数必须传。

### 3.4.4 更极致的方案：`**kwargs` 完全动态工具

如果工具种类非常多（比如对接了 20+ 个微服务），可以为每个服务写一个完全通用的透传工具：

```python
from typing import Any
from pydantic import BaseModel, Field


class ServiceCallInput(BaseModel):
    service: str = Field(description="服务名称：order / inventory / logistics / notification / billing")
    method: str = Field(description="方法名。如 order 服务支持: get_by_id, create, cancel, list_by_user")
    params: dict[str, Any] = Field(default_factory=dict, description="方法参数，键值对形式")


@tool(args_schema=ServiceCallInput)
async def service_call(service: str, method: str, params: dict[str, Any]) -> str:
    """
    通用微服务调用工具。
    
    可用服务及方法：
    - order: get_by_id(需要order_id), create(需要data), cancel(需要order_id), list_by_user(可选keyword)
    - inventory: check(需要product_id), adjust(需要product_id,quantity), list()
    - logistics: track(需要order_id), estimate(需要order_id)
    - notification: send(需要user_id,message,channel)
    - billing: get_balance(需要user_id), charge(需要user_id,amount,description)
    """
    try:
        service_func = SERVICE_REGISTRY.get(service)
        if not service_func:
            return f"服务 '{service}' 不存在。可用：{list(SERVICE_REGISTRY.keys())}"
        
        method_func = getattr(service_func, method, None)
        if not method_func:
            return f"方法 '{service}.{method}' 不存在。可用方法见工具描述。"
        
        return await method_func(**params)
    except Exception as e:
        return f"服务调用失败 [{service}.{method}]：{e}。请勿重试，告知用户操作失败。"
```

**但这种方式有个代价：** `params: dict` 没有结构化的 JSON Schema，LLM 在填充参数时容易出错。所以只在工具数量过多（如 30+）时考虑，一般情况下用 3.4.2 的 `action` 枚举方案就足够了。

## 3.5 意图识别：到底要不要调工具

### 3.5.1 意图识别为什么重要

看这两句话：

> 用户 A："帮我查一下我昨天的订单" → 需要调工具
> 用户 B："我想了解一下什么是订单号" → 不需要调工具

如果 Agent 对 B 也调了 `search_order`，返回一个空结果，然后 LLM 看着空结果强行编造回复——用户体验极差。

**意图识别的目标：在调用工具之前，先判断"用户是在要求做事，还是在聊天/提问"。**

### 3.5.2 方案一：靠 LLM 原生判断（`tool_choice="auto"`）

大多数情况下，GPT-4o / Claude 等强模型自带的判断能力已经很好了。写清楚工具描述，LLM 会在该调的时候调，不该调的时候不调。

```python
# 工具描述写好了，LLM 自己会判断
@tool
async def search_order(order_id: str) -> str:
    """
    查询指定订单号的订单详情。仅在用户明确提到一个订单号并要求查询时使用。
    如果用户只是在聊订单相关的一般性话题，不要调用此工具。
    """
    ...
```

但单靠 LLM 判断有两个风险：
- 弱模型（如 4o-mini）可能判断失误
- 边界模糊的场景（用户说"你觉得我应该取消这个订单吗"——这是要调工具查订单状态，还是纯聊天给建议？）

### 3.5.3 方案二：前置意图分类（双阶段路由）

用一次轻量级 LLM 调用先做意图分类，再决定走工具调用路径还是纯对话路径：

```python
# core/intent_classifier.py
from enum import Enum
from pydantic import BaseModel, Field
from langchain_core.prompts import ChatPromptTemplate
from core.llm import call_llm_with_retry


class Intent(str, Enum):
    ACTION = "action"           # 需要执行操作（查订单、发邮件、改设置）
    QUESTION = "question"        # 知识性问题（什么是XX、怎么理解XX）
    CHITCHAT = "chitchat"        # 闲聊（你好、谢谢、今天心情怎么样）
    CLARIFICATION = "clarification"  # 用户消息不明确，需要追问


class IntentResult(BaseModel):
    intent: Intent
    confidence: float = Field(ge=0.0, le=1.0)
    reasoning: str = Field(default="")


INTENT_PROMPT = ChatPromptTemplate.from_messages([
    ("system", """你是一个意图分类器。分析用户消息，判断对话意图。

意图类型：
- action：用户要求执行某个具体操作（查询、创建、修改、删除、发送）。这些操作需要调用工具才能完成。
  示例："帮我查订单 ORD-001"、"取消我的订阅"、"把名字改成张三"
  
- question：用户在询问知识、概念、原理、方法。不需要调用工具，用模型自身知识回答即可。
  示例："什么是 RESTful API？"、"Python 怎么处理并发？"、"你推荐什么编程语言？"
  
- chitchat：纯社交对话，不需要工具也不需要专业知识。
  示例："你好"、"谢谢"、"今天星期几"、"讲个笑话"
  
- clarification：用户消息太模糊，无法判断意图，需要追问。
  示例："那个东西怎么样了"、"帮我看一下"（没有指明看什么）

只输出 JSON。"""),
    ("user", "{message}"),
])


async def classify_intent(message: str) -> IntentResult:
    """对用户消息做意图分类"""
    prompt = INTENT_PROMPT.format(message=message)
    
    response = await call_llm_with_retry(
        messages=prompt.to_messages(),
        model="gpt-4o-mini",          # 意图分类用小模型即可
        response_format={"type": "json_object"},
    )
    
    return IntentResult(**json.loads(response))
```

路由层根据意图分派：

```python
# routes/chat.py
from core.intent_classifier import classify_intent, Intent


@router.post("/completions")
async def chat_completions(request: ChatRequest):
    # 第一步：意图分类
    intent_result = await classify_intent(request.message)
    logger.info("意图分类 | session_id=%s | intent=%s | confidence=%.2f",
                request.session_id, intent_result.intent, intent_result.confidence)

    # 第二步：按意图路由
    if intent_result.intent == Intent.ACTION and intent_result.confidence > 0.7:
        # 绑定工具，走 Agent 工具调用流程
        return await _handle_with_tools(request)
    
    elif intent_result.intent == Intent.CLARIFICATION:
        # 追问用户，不走工具
        return await _handle_clarification(request, intent_result.reasoning)
    
    else:
        # 纯对话，不绑定工具
        return await _handle_chat_only(request)
```

### 3.5.4 方案三：轻量级关键词匹配兜底

对于非常明确的指令型表述，用关键词做第一层快速判断，可以省掉一次 LLM 调用：

```python
# core/intent_quick_check.py
import re

# 强烈的"要求执行操作"的关键词模式
ACTION_PATTERNS = [
    r"(帮|给|替)(我|我们).*(查|搜|找|取消|删除|修改|创建|发送|下单|订|买)",
    r"(查一下|搜一下|看一下|找一下|帮我查|我要查).*",
    r"^(取消|删除|修改|创建|发送|下单|订购|购买|设置|切换)",
]

# 纯闲聊关键词
CHITCHAT_PATTERNS = [
    r"^(你好|hi|hello|嗨|在吗|谢谢|多谢|再见|bye|晚安|早安)",
    r"^(讲个|来个|说个)(笑话|故事|段子)",
]


def quick_intent_check(message: str) -> str | None:
    """
    快速关键词意图检查。
    如果不能确定，返回 None，交给 LLM 判断。
    """
    message = message.strip()
    
    for pattern in ACTION_PATTERNS:
        if re.search(pattern, message):
            return "action"
    
    for pattern in CHITCHAT_PATTERNS:
        if re.search(pattern, message):
            return "chitchat"
    
    return None  # 不确定，交给 LLM


# 使用
quick = quick_intent_check(request.message)
if quick == "action":
    intent_result = IntentResult(intent=Intent.ACTION, confidence=0.9)
elif quick == "chitchat":
    intent_result = IntentResult(intent=Intent.CHITCHAT, confidence=0.95)
else:
    intent_result = await classify_intent(request.message)  # LLM 判断
```

### 3.5.5 三个方案的对比与组合

```
用户消息到达
     │
     ▼
┌──────────────┐
│ 关键词快速判断 │  ← 方案三：几乎零延迟，覆盖 30% 的明确场景
└──┬───────┬───┘
   │       │
  明确    不确定
   │       │
   ▼       ▼
  直接    ┌──────────────┐
  路由    │ LLM 意图分类  │  ← 方案二：1~2 秒，覆盖剩余 70%
          └──┬───────┬───┘
             │       │
          action   其他
             │       │
             ▼       ▼
         ┌──────┐ ┌──────┐
         │Agent │ │纯对话 │  ← 方案一：Agent 内部 LLM 再次判断具体调哪个工具
         │+工具  │ │模式   │
         └──────┘ └──────┘
```

**推荐组合：方案三（关键词兜底）+ 方案二（LLM 分类）。** 方案一（纯靠 LLM）作为 Agent 内部的基础能力。三层叠加，准确率最高，延迟增量最小。

### 3.6.2 工具注册中心

### 3.6.3 按场景限缩工具集

不是所有场景都需要全量工具。把工具分组，不同 Agent 节点只绑定所需的子集：

```python
# core/agent.py
from langgraph.graph import StateGraph, MessagesState
from langgraph.prebuilt import ToolNode
from langchain_openai import ChatOpenAI
from tools import ORDER_TOOLS, INVENTORY_TOOLS, GENERAL_TOOLS

llm = ChatOpenAI(model="gpt-4o")

def build_agent():
    graph = StateGraph(MessagesState)

    # 意图路由节点：只绑通用工具
    llm_general = llm.bind_tools(GENERAL_TOOLS)

    # 订单处理节点：只绑订单工具
    llm_order = llm.bind_tools(ORDER_TOOLS)
    order_tool_node = ToolNode(ORDER_TOOLS)

    # 库存处理节点：只绑库存工具
    llm_inventory = llm.bind_tools(INVENTORY_TOOLS)
    inventory_tool_node = ToolNode(INVENTORY_TOOLS)

    # ... 定义节点和边 ...
```

这样每个节点的工具定义 token 开销大幅降低，LLM 选择工具的准确率也大幅提高。

## 3.8 常见踩坑清单

| 坑 | 现象 | 原因 | 解法 |
|----|------|------|------|
| 工具反复重试失败 | LLM 连续 5 次调同一个工具，每次报同样的错 | 工具抛异常，LLM 看到 Exception 就本能重试 | 工具返回文本错误信息，明确写"请勿重试" |
| 工具被误调 | 用户聊天说"帮我推荐一本书"，Agent 跑去调 search_order | 工具描述太宽泛，LLM 产生幻觉性匹配 | 工具描述写"仅在用户明确提到订单号时调用" |
| 工具参数填错 | LLM 传了 `"明天"` 给要 `YYYY-MM-DD` 的参数 | LLM 不严格按格式填参数 | 工具内部做参数容错（`"明天"` → `date+1`） |
| 工具太多导致选错 | 20 个工具里 LLM 选了错误的一个 | 工具过多，描述相似时 LLM 容易混淆 | 合并为通用工具 + action 枚举，按场景分组 |
| 工具返回值太长 | Agent 拿到 5000 字结果后回复跑题 | 大段文本淹没了 LLM 的注意力 | 工具做截断（≤1500 字符），或返回引用 key |
| 该调工具却没调 | LLM 用自己的知识编了一个答案，但实际需要查数据库 | tool_choice="auto" 时 LLM 过于自信 | 加意图分类步骤，对 action 类意图强制绑工具 |
| 慢模型判断意图太慢 | 每次请求要多等 1~2 秒做意图分类 | 意图分类走了大模型 | 用 4o-mini 做分类，或关键词快速判断走捷径 |

## 3.9 本章小结

| 要点 | 核心做法 | 一句话 |
|------|---------|--------|
| 工具定义 | `@tool` 装饰器 + Pydantic args_schema | 一个函数 = 一个工具，参数类型自动生成 Schema |
| 工具绑定 | `llm.bind_tools(tools, tool_choice="auto")` | 工具定义随每次请求发送，token 开销值得关注 |
| 防止重复调用 | 工具返回文本报错 + tool_guard 检测 + recursion_limit | 三层防线：工具自己说"别重试" > 守卫拦截 > 步数上限 |
| 通用工具 `**kwargs` | 一个系统一个工具，用 `action: Literal[...]` 分发 | 3 个工具代替 15 个，token 省了，准确率高了 |
| 意图识别 | 关键词快速判断 → LLM 分类 → 路由分派 | 不能全靠 LLM 的 tool_choice="auto"，关键路径要前置判断 |
| 工具容错 | 参数容错 + 异常转文本 + 返回值截断 | 工具在任何输入下都应该返回可读字符串，不能崩 |

**工具集的设计原则：工具是 LLM 的眼睛和手——眼睛要看得准（意图识别），手要稳（容错健壮），而且不能用一把锤子去拧螺丝（工具分组，场景限缩）。**
