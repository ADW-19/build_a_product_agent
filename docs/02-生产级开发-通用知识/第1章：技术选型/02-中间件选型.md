# 第二章：中间件选型

> **备注**：本章针对生产环境做完整规划。DEMO 阶段只需了解"为什么这么选"，不需要全部部署——可以先上 PostgreSQL + Redis，其余在代码中预留接口即可。(**如果要写系统设计书这一章非常重要！！！务必体现你的设计思路在说明书中**))

## 2.1 中间件全景图

```
                          ┌──────────────┐
                          │   用户请求    │
                          └──────┬───────┘
                                 │
                          ┌──────▼───────┐
                          │   FastAPI    │
                          └──┬──┬──┬──┬──┘
                             │  │  │  │
          ┌──────────────────┘  │  │  └──────────────────┐
          │                     │  │                     │
    ┌─────▼─────┐    ┌─────────▼──▼───┐    ┌────────────▼──────┐
    │ PostgreSQL │    │    Redis       │    │     RabbitMQ       │
    │  业务数据  │    │  缓存 / 限流   │    │   异步任务队列      │
    └───────────┘    └───────────────┘    └───────────────────┘
                             │
                     ┌───────▼───────┐
                     │    Milvus     │
                     │   向量检索    │
                     └───────────────┘
```

四个中间件各管一摊：**PostgreSQL 存业务数据，Milvus 做向量检索，Redis 扛高并发读写，RabbitMQ 削峰填谷**。

## 2.2 PostgreSQL vs MySQL 8.0

### 2.2.1 结论

**选 PostgreSQL，因为 Agent 场景天然需要非结构化数据能力 + 向量扩展。**

### 2.2.2 什么是关系型数据库？（概念）

关系型数据库就是"用表格存数据"的数据库。每张表有固定的列结构（Schema），数据一行一行存，不同表之间通过"外键"关联。——这是最经典的企业数据存储方式，MySQL 和 PostgreSQL 都属于这一类。

### 2.2.3 核心对比

| 维度      | MySQL 8.0                               | PostgreSQL                                            |
| --------- | --------------------------------------- | ----------------------------------------------------- |
| JSON 查询 | ⚠️ JSON_EXTRACT，函数式语法，不够直观 | ✅`->>` 和 `@>` 操作符，JSONB 原生索引            |
| 全文搜索  | ⚠️ ngram 分词仅限中日韩，英文一般     | ✅ 内置 `tsvector` + `tsquery`，支持多语言分词    |
| 向量支持  | ❌ 原生不支持                           | ✅`pgvector` 扩展，可在同一张表做混合检索           |
| 扩展能力  | 弱，插件生态有限                        | 强，`pgvector`、`pg_cron`、`postgis` 等活跃扩展 |
| 复杂查询  | ⚠️ 嵌套查询优化一般，CTE 较晚才完善   | ✅ 优化器成熟，CTE + 窗口函数 + LATERAL JOIN          |
| 并发模型  | 单进程多线程                            | 多进程，长事务更稳                                    |

### 2.2.4 三个决定性的理由

**理由一：Agent 大量存储 JSON。**

Agent 的每一步推理、工具调用结果、用户上下文都是半结构化的。比如一条对话记录：

```json
{
  "session_id": "abc123",
  "messages": [
    {"role": "user", "content": "帮我查一下订单"},
    {"role": "assistant", "tool_calls": [{"name": "search_order", "args": {"user_id": 42}}]},
    {"role": "tool", "name": "search_order", "content": "[{...}]"}
  ]
}
```

PostgreSQL 的 **JSONB**（二进制 JSON）支持在 JSON 字段上建 GIN 索引，可以直接在 WHERE 子句里查 JSON 内部字段，速度不输普通列。MySQL 的 JSON 功能是"能用"级别，PostgreSQL 是"好用"级别。

**理由二：Agent 需要"模糊搜索 + 精确过滤"混合查询。**

例如："找出最近一周用户提到'退款'的对话，按时间倒序"——这既需要全文搜索（`tsvector` 查"退款"），又需要精确过滤（时间范围 + 用户 ID）。PostgreSQL 一条 SQL 搞定，MySQL 要把全文搜索交给 Elasticsearch 或其他外部组件，架构复杂度立刻翻倍。

**理由三：pgvector 让向量和业务数据同库共存。**

Agent 的 RAG（检索增强生成）场景需要在同一个查询里混合向量相似度排序和业务字段过滤。PostgreSQL + `pgvector` 扩展可以做到：

```sql
SELECT content, 1 - (embedding <=> %s) AS similarity
FROM documents
WHERE user_id = $1 AND created_at > $2
ORDER BY embedding <=> %s
LIMIT 10;
```

MySQL 做不到这一点，因为根本没有向量索引能力。如果用 MySQL，你无论如何都要额外引入一个向量数据库。

### 2.2.5 什么时候选 MySQL？

如果你的业务是**纯 OLTP，表结构固定、无 JSON 查询/全文检索/向量需求**，MySQL 仍然是成熟的选择。但对于 Agent 后端这些需求全是刚需，PostgreSQL 更合适。

## 2.3 Milvus vs Chroma vs FAISS

### 2.3.1 结论

**选 Milvus，因为它是面向生产的分布式向量数据库。Chroma 适合原型开发，FAISS 只是一个索引库。**

### 2.3.2 什么是向量检索？（概念）

LLM 不认识你的私有数据（比如你公司的产品手册、用户历史对话），但它可以先把文本"翻译"成一串数字——这叫**向量嵌入（embedding）**。两段意思相近的文本，它们的向量在数学空间里距离就近。

"帮我查一下退货政策" → `[0.12, -0.34, 0.56, ..., 0.78]`（1024 维浮点数向量）

当你提问时，系统把问题也转成向量，然后在库里找到距离最近的几条文本，塞进 LLM 的 prompt 里作为上下文——这就是 **RAG（检索增强生成）**。向量数据库就是专门干"找最近向量"这件事的。

### 2.3.3 三者的本质区别

```
              Chroma              FAISS               Milvus
             (向量数据库)        (向量索引库)        (分布式向量数据库)
             
本质：     轻量级嵌入式 DB       Facebook 开源       Zilliz 开源的
           一键 pip install    的向量检索库        分布式系统

部署：     pip install即可      无独立进程，        独立服务，需要
          无独立服务             只是一段 Python 代码  docker/k8s 部署

适用：     原型/DEMO           本地数据集          生产环境
           (<10万条向量)        (<100万条)          (亿级向量)
```

**FAISS 不是数据库。**

FAISS 不提供数据持久化、不提供 CRUD、不提供分布式部署、不提供权限控制。它就是一堆 C++ 实现的向量索引算法，你得自己管理数据存哪、怎么增删改查。把它类比：FAISS 是"手搓的排序算法"，Milvus 是"数据库管理系统"。

**Chroma 不适合高并发 C 端。**

Chroma 设计目标是"开发体验优先"——pip install 零配置就能跑，非常适合 DEMO。但它：

- 不支持真正的水平扩展（不能像 Milvus 那样加节点扩容）
- 索引算法单一，海量数据下召回率和速度都不如 Milvus
- 缺乏多租户隔离、资源配额等生产特性

### 2.3.4 核心对比

| 维度         | FAISS               | Chroma        | Milvus           |
| ------------ | ------------------- | ------------- | ---------------- |
| 部署模式     | 嵌入代码            | 嵌入/独立进程 | 分布式集群       |
| 持久化       | ❌ 需手写           | ✅ 半自动     | ✅ 原生支持      |
| 水平扩展     | ❌                  | ❌            | ✅ 分片+副本     |
| 索引类型     | 多种（IVF/HNSW/PQ） | 有限          | 丰富（11种+）    |
| 混合检索     | ❌                  | ⚠️ 实验性   | ✅ 标量过滤+向量 |
| 一致性保证   | 无                  | 弱            | 强一致性可选     |
| 数据规模上限 | ~100 万             | ~100 万       | 亿级+            |
| QPS 上限     | 受限于单机          | 受限于单机    | 数千 QPS         |

### 2.3.5 三个决定性的理由

**理由一：高并发 C 端需要分布式能力。**

C 端场景下，同时可能有数千用户在做 RAG 问答。Milvus 支持分片（把一个集合的数据拆到多台机器）和副本（同一份数据存多份），既扛高 QPS，又保证高可用。Chroma 和 FAISS 都是单机方案，一台机器撑不住就只能改代码。

**理由二：混合检索（标量 + 向量）是 Agent 的刚需。**

Agent 的知识库检索往往带过滤条件：

- "只搜索我团队内部的文档"（tenant_id 过滤）
- "只搜最近 30 天更新的产品手册"（时间过滤）
- "只搜状态为'已发布'的知识条目"（状态过滤）

Milvus 支持在向量检索的同时做标量过滤，且分区键（partition key）可以显著加速这类过滤。Chroma 的过滤功能是附加的，性能不可比。

**理由三：pgvector 不够用的时候，Milvus 是唯一出口。**

你可能会问：PostgreSQL + pgvector 不是也能做向量检索吗？对，**10 万条以内 pgvector 完全够用**。但到了百万级，pgvector 的 IVF/IVFFlat 索引精确度开始下降，而且它没有分布式能力——PostgreSQL 是一个单体数据库，你不可能把一张表的向量拆分到多台机器上。当数据量上去了，Milvus 是必经之路。

## 2.4 为什么需要 Redis

### 2.4.1 结论

**Redis 在高并发系统中承担三个角色：缓存加速、分布式锁、令牌桶限流。**

### 2.4.2 什么是 Redis？（概念）

Redis 是一个**基于内存的键值存储系统**。数据存在内存而非磁盘，所以读写极快（微秒级）。它支持字符串、列表、哈希、集合、有序集合等多种数据结构，还支持设置过期时间、发布订阅、Lua 脚本等高级能力。

### 2.4.3 Redis 在这个 Agent 系统中的四个具体用途（其实不止，还有session_id存储与区分等，可以具体问AI）

#### ① 对话缓存

用户和 Agent 的对话上下文每次请求都要读取，直接查 PostgreSQL 太慢：

```
请求 → Redis（命中，0.5ms）→ 返回  ✅
请求 → PostgreSQL（未命中，8ms）→ 写入 Redis → 返回  ⚠️
```

C 端场景下，同一用户可能在短时间内连续发送多条消息，缓存命中率极高。Redis 把 P99 延迟从几十毫秒压到个位数毫秒。

#### ② Token 限流（令牌桶）

高并发下必须限制每个用户调用 LLM 的频率，避免 API 费用爆炸：

```
Key: rate_limit:user:10086
Value: {"tokens": 50, "last_refill": "2026-05-09T10:00:00Z"}
TTL: 60s
```

Redis 的原子性操作（`INCR`、`EXPIRE`、Lua 脚本）天然适合做令牌桶计数器。每次请求到达时：

1. 检查当前令牌数
2. 如果 > 0，减 1 放行
3. 如果 = 0，返回 429（Too Many Requests）

这个过程全程在 Redis 内存中完成，单次检查 < 1ms。

#### ③ 分布式锁

Agent 的某些操作（比如：同一个 session 的对话不能并发处理；定时任务不能多实例重复执行）需要锁：

```python
# 在 Redis 中设置一个带过期时间的 key 作为锁
lock_key = "lock:session:abc123"
# SET lock_key random_value NX EX 10
# 拿到锁的执行，拿不到的等待或跳过
```

`NX` 表示"key 不存在才创建"，`EX 10` 表示 10 秒后自动释放，防止死锁。

#### ④ 短时间窗口统计（滑动窗口）

C 端需要统计"过去 1 分钟内某接口被调用了多少次"用于熔断：

```
Key: window:agent_chat:1715251200（Unix 分钟时间戳）
INCR → 返回当前计数
TTL: 120s（自动清理）
```

Redis 的 `INCR` + `EXPIRE` 组合是计数器场景的事实标准解决方案。

### 2.4.4 不用 Redis 会怎样？

| 场景     | 无 Redis                   | 有 Redis               |
| -------- | -------------------------- | ---------------------- |
| 对话读取 | 每次查 PostgreSQL（8ms）   | 缓存命中时 0.5ms       |
| 限流     | 查 DB 做计数（慢且不准）   | 内存原子操作（准且快） |
| 分布式锁 | 用数据库表模拟（死锁风险） | SET NX EX 天然支持     |
| 窗口统计 | 查 DB COUNT（大表扫描）    | 内存计数器实时更新     |

## 2.5 为什么需要 RabbitMQ（消息队列）

### 2.5.1 结论

**消息队列解决三个问题：削峰、解耦、异步。Agent 的 LLM 调用是慢操作，必须用队列做生产-消费模式。**

### 2.5.2 什么是消息队列？（给大学生）

消息队列是一种**生产者-消费者**模式的中间件。生产者把任务消息扔到队列里就完事了，不用等任务真正执行完。消费者从队列里取任务来处理，处理完再通知结果。

```
[生产者 FastAPI] ──发送消息──▶ [RabbitMQ 队列] ──消费──▶ [Worker 进程]
   ↓ 立刻返回 "处理中"                                       ↓ 慢慢调 LLM
```

### 2.5.3 RabbitMQ 在这个 Agent 系统中的四个具体用途

#### ① 异步 Agent 执行（削峰）

一次 Agent 调用可能包括多轮 LLM 推理 + 多次工具调用，耗时 3~30 秒不等。HTTP 请求不可能同步等这么久。正确姿势：

```
用户发送消息
   │
   ▼
FastAPI 接收请求 → 写入 RabbitMQ → 立即返回 "ok"（200ms 内）
                                             │
                                      ┌──────▼───────┐
                                      │  Worker 进程   │
                                      │  Agent 执行中  │  ← 耗时 3~30s
                                      │  结束后通知前端 │
                                      └──────────────┘
```

当并发请求瞬间从 100 飙到 10000，消息队列在中间当缓冲——所有请求先进队列排队，Worker 按自己的节奏消费，后端不会被打崩。

#### ② 服务解耦

Agent 执行结束后可能有多个后续动作：保存对话记录、更新用户统计数据、触发通知推送。用消息队列做事件驱动，Agent Worker 只负责"发事件"，具体谁消费它不管：

```
Agent 执行完成
   │
   ▼
发布 "agent.run.completed" 事件到 Exchange
   │
   ├──▶ 队列 A → 消费者：记录对话日志到 PostgreSQL
   ├──▶ 队列 B → 消费者：更新用户配额
   └──▶ 队列 C → 消费者：异步通知（WebSocket 推送给前端）
```

每个消费者都是独立进程，挂了不影响其他逻辑。新增消费者也不需要改 Agent Worker 的代码。

#### ③ 失败重试

LLM 调用可能因网络抖动、API 限流等原因失败。消息队列天然支持：

- 消费失败 → NACK → 消息重回队列 → 自动重试
- 重试 N 次仍失败 → 转入死信队列（Dead Letter Queue）→ 人工排查

这些重试逻辑如果自己手写，代码量会很大且容易出 bug。

#### ④ 优先级调度

C 端场景下，付费用户的 Agent 请求应该优先于免费用户：

```
RabbitMQ 优先级队列：
  VIP 用户消息   → priority=10 → 优先消费
  普通用户消息   → priority=0  → 正常排队
```

### 2.5.4 RabbitMQ vs Kafka vs Redis Pub/Sub

| 维度            | RabbitMQ      | Kafka              | Redis Pub/Sub                |
| --------------- | ------------- | ------------------ | ---------------------------- |
| 消息确认（ACK） | ✅ 原生支持   | ✅ 支持            | ❌ 不支持（fire-and-forget） |
| 死信队列        | ✅ 原生支持   | ⚠️ 需手动实现    | ❌ 不支持                    |
| 优先级队列      | ✅ 原生支持   | ❌ 不支持          | ❌ 不支持                    |
| 消息持久化      | ✅            | ✅                 | ❌                           |
| 吞吐量          | 万级/秒       | 百万级/秒          | 十万级/秒                    |
| 适用场景        | 任务分发、RPC | 日志流、大数据管道 | 即时通知（丢了也无所谓）     |

**Agent 的消息模型是"任务分发"而非"数据流"**——每条消息代表一个需要执行的任务，每条都要保证被处理、允许失败重试、支持优先级。这些是 RabbitMQ 的传统强项。Redis Pub/Sub 不持久化、无 ACK，消息丢了就丢了——C 端用户发了一条消息结果丢了，这是不可接受的。

Kafka 适合"日志采集、事件溯源"这类海量数据流场景，但在"一对一任务分发 + 失败重试 + 优先级"的场景下，RabbitMQ 更合适。

## 2.6 四者协作全景

以一个典型的用户对话为例，展示四个中间件如何协作：

```
用户： "帮我查一下最新的退货政策"
   │
   ▼
┌──────────────────────────────────────────────────┐
│ ① FastAPI 收到请求                                │
│    Redis 检查限流 → 通过                          │
│    Redis 检查该 session 是否有互斥锁 → 无锁       │
│    消息写入 RabbitMQ → 返回 "处理中"              │
└──────────────────┬───────────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────────┐
│ ② Worker 从 RabbitMQ 取出任务                     │
│    Redis 加锁（lock:session:xxx）                 │
│    Redis 读取对话缓存 → 命中                      │
└──────────────────┬───────────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────────┐
│ ③ Agent 执行（LangGraph）                         │
│    LLM 判断：需要检索知识库 → 调用 search_docs    │
│    search_docs 工具：将问题转为向量               │
│    Milvus 向量检索 + tenant_id 过滤               │
│    返回匹配的文档片段                             │
└──────────────────┬───────────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────────┐
│ ④ LLM 生成最终回复                                │
│    prompt = 用户问题 + Milvus 检索结果 + 对话历史  │
│    LLM → 生成回复                                 │
└──────────────────┬───────────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────────┐
│ ⑤ 结果回写                                       │
│    PostgreSQL 保存完整对话记录                     │
│    Redis 更新对话缓存 + 释放锁                    │
│    RabbitMQ 发布 "agent.run.completed" 事件       │
│    → 通知前端（WebSocket）                        │
│    → 更新用户配额                                 │
└──────────────────────────────────────────────────┘
```

## 2.7 DEMO vs 生产 的部署差异

| 中间件     | DEMO 阶段                   | 生产阶段                       |
| ---------- | --------------------------- | ------------------------------ |
| PostgreSQL | Docker 单实例               | 主从复制 + 连接池（PgBouncer） |
| Milvus     | pgvector 替代（10万向量内） | Milvus 集群（分片 + 副本）     |
| Redis      | Docker 单实例               | Redis Sentinel / Cluster       |
| RabbitMQ   | 不需要（同步执行即可）      | 集群 + 镜像队列 + 死信队列     |

## 2.8 本章小结

四个中间件各有不可替代的职责：

- **PostgreSQL**：存业务数据 + JSON 查询 + 全文搜索 + 向量检索（10万内），替代 MySQL 的理由是 JSONB / tsvector / pgvector 三个能力刚好是 Agent 刚需。
- **Milvus**：海量向量检索（百万级+），替代 Chroma/FAISS 的理由是分布式、混合检索、生产级稳定性。
- **Redis**：缓存加速 + 限流 + 分布式锁 + 窗口统计，高并发下保护数据库和 LLM API 不被击穿。
- **RabbitMQ**：异步任务 + 削峰 + 失败重试 + 服务解耦，解决"LLM 调用太慢不能同步返回"这个 Agent 后端最核心的架构问题。
