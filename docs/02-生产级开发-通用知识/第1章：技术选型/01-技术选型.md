# 第一章：技术选型

## 1.1 技术栈概览

| 组件       | 选型            | 版本            |
| ---------- | --------------- | --------------- |
| 编程语言   | Python          | 3.11            |
| Web 框架   | FastAPI         | ≥0.115         |
| Agent 编排 | LangGraph       | ≥0.3           |
| LLM 工具层 | LangChain       | ≥1.0（稳定版） |
| 数据验证   | Pydantic        | v2              |
| 异步运行时 | uvicorn + anyio | -               |

## 1.2 这个搭配在工业界正确吗？

**结论：正确，且是当前（2025-2026）的主流推荐组合。**

几个事实支撑：

1. **LangChain 1.0 的发布标志着 API 成熟稳定**。2025 年 LangChain 正式发布 1.0 稳定版，核心接口（ChatModel、Tool、PromptTemplate、Message 体系）进入长期稳定承诺，不再有 0.x 时代频繁的 breaking change。官方明确 LangGraph 是多 Agent、有状态编排的首选方案，LangChain 定位为"模型调用 + 工具定义"的基础层。
2. **FastAPI 是 Python 异步 Web 框架的事实标准**。LangGraph 的 Agent 执行是异步的（`ainvoke`、`astream`），FastAPI 原生 async 支持与 LangGraph 的运行模型天然匹配，不存在事件循环冲突。
3. **Pydantic v2 是共同依赖**。LangChain 1.0、LangGraph、FastAPI 三方都基于 Pydantic v2 做数据序列化和校验，类型体系统一，无适配成本。
4. **工业界案例**：Elastic、Airbnb、Replit、多家 AI 初创公司的 Agent 后端均公开使用了 FastAPI + LangGraph 的组合；LangChain 官方自身发布的模板项目（如 `langgraph/examples`）也采用 FastAPI 做 API 服务。

如果非要挑一个需要留意的点：**LangChain 包体积较大、抽象层较厚**。在简单场景下可以只用 LangGraph + 直接调模型 SDK（如 `openai` 包），但既然你要用 LangChain 的 tool 定义、prompt template、output parser 等能力，那就没有多余成本。

## 1.3 为什么选 Python 3.11

- **asyncio 性能提升**：3.11 对 asyncio 的任务调度和异常处理有显著优化，高并发 Agent 场景收益直接。
- **异常组（ExceptionGroup）**：3.11 引入的 `ExceptionGroup` 让并发 Agent 任务的异常聚合更干净，LangGraph 内部已在利用这一特性。
- **tomllib 标准库**：配置文件可以用标准库读取，不需要额外依赖 `toml` 包。
- **生态兼容性**：截至 2026 年初，LangChain 1.0 / LangGraph / FastAPI 对 3.11 的支持最成熟。3.12 也在追赶，但部分依赖（如 `grpcio`）的预编译 wheel 在 3.12 上偶有问题。

## 1.4 为什么选 LangChain 1.0 稳定版

LangChain 从 0.x 跨越到 1.0 带来的关键变化：

### 1.0 之前（0.x 时代）的问题

- API 不稳定，版本间频繁 breaking change
- 过度抽象（LCEL、Runnable 体系对新手心智负担重）
- 包体积膨胀，`langchain` 包耦合了大量社区插件

### 1.0 的改进

- **接口稳定承诺**：核心模块（`langchain-core`）进入语义化版本管理，不再随意破坏 API。`ChatModel.invoke()`、`BaseTool`、`BaseMessage` 等关键接口已冻结。
- **包拆分明确**：`langchain-core`（核心抽象）、`langchain`（高级封装）、`langchain-openai`（OpenAI 集成）、`langchain-community`（社区贡献）各司其职，安装时按需引入。
- **与 LangGraph 的协作模式成熟**：1.0 后官方确立了"LangChain 提供工具和模型调用，LangGraph 负责状态编排"的分工。两边的接口配合已经过大量生产验证。
- **向后兼容保证**：1.x 主版本内不会引入 breaking change，这对工业项目至关重要。

### 何时可以不用 LangChain

如果你只用一个模型供应商（比如只用 OpenAI），且不需要复杂的 prompt 模板、output parser、社区工具生态，那可以只用 `openai` + `langgraph`，无需引入 LangChain。但多模型切换、工具定义（`@tool` 装饰器）、标准化消息格式这些需求一旦出现，LangChain 就是最省力的选择。

## 1.5 为什么选 FastAPI

### 对比 Flask

| 维度              | FastAPI                          | Flask                   |
| ----------------- | -------------------------------- | ----------------------- |
| 异步原生支持      | ✅ async/await 一等公民          | ❌ 需要扩展（quart 等） |
| 自动 OpenAPI 文档 | ✅ 内置 swagger / redoc          | ❌ 需要插件             |
| 请求/响应校验     | ✅ Pydantic v2 原生集成          | ❌ 需要 marshmallow 等  |
| 流式响应（SSE）   | ✅`StreamingResponse` 直接支持 | ⚠️ 需要额外处理       |
| 性能              | 异步 + uvloop，接近 Node.js      | 同步模式，多线程        |

Agent 后端的核心场景——**流式输出（SSE/WebSocket）、长时间运行的 Agent 任务、高并发工具调用**——FastAPI 全部原生支持，Flask 则需要大量额外工作。

### 对比 Django / Django-Ninja

Django 适合全栈 Web 应用，做纯 API + Agent 场景太重了。ORM、模板引擎、中间件体系在纯 Agent 后端中有 80% 用不到。Django-Ninja 缩小了差距，但社区体量仍远小于 FastAPI。

### 对比纯 gRPC

gRPC 对 Agent 后端不是好选择。Agent 的输入输出以自然语言为主，可调试性比传输效率更重要。REST + SSE 的 HTTP 方案在开发效率和可观测性上明显胜出。如果未来内部服务间有强类型通信需求，可以在 FastAPI 之上额外暴露 gRPC 端口。

## 1.6 LangChain vs LangGraph 的职责划分

这是最容易混淆的地方。一句话总结：

> **LangChain 管"调用模型"，LangGraph 管"编排流程"。**

```
┌─────────────────────────────────────┐
│           LangGraph                  │
│  ┌──────┐   ┌──────┐   ┌──────┐    │
│  │ Node │──▶│ Node │──▶│ Node │    │
│  │  A   │   │  B   │   │  C   │    │
│  └──────┘   └──────┘   └──────┘    │
│     │           │           │       │
│     ▼           ▼           ▼       │
│  ┌──────────────────────────────┐   │
│  │        LangChain 1.0          │   │
│  │  • ChatModel（模型调用）      │   │
│  │  • Tool（工具定义）           │   │
│  │  • PromptTemplate（提示词）   │   │
│  └──────────────────────────────┘   │
└─────────────────────────────────────┘
```

**LangChain 做的事：**

- 统一的模型调用接口（`ChatOpenAI`、`ChatAnthropic` 等）
- 工具定义和 Schema 生成（`@tool` 装饰器，Pydantic 参数）
- Prompt 模板管理
- 消息格式标准化（SystemMessage、HumanMessage、AIMessage、ToolMessage）
- 输出解析器（JSON、结构化提取等）

**LangGraph 做的事：**

- 有状态图（StateGraph）—— Agent 执行的状态管理
- 条件边（conditional edges）—— 根据 LLM 输出决定下一步
- 循环和分支 —— 工具调用循环、多步推理
- 检查点（checkpointer）—— 持久化执行状态，支持断点续跑和回溯
- 人工介入（human-in-the-loop）—— 在关键节点暂停等待人工审批
- 流式输出 —— `astream_events` 支持逐 token 推送

## 1.8 本章小结

FastAPI + LangGraph + LangChain 1.0 + Python 3.11 是 2025-2026 年工业界开发 Agent 后端的最主流、经过充分验证的技术组合。LangChain 1.0 解决了 0.x 时代 API 不稳定的核心痛点，进入了长期维护的稳定轨道。三者都基于 Pydantic v2 和 asyncio，类型体系统一，异步模型一致，没有适配成本。
