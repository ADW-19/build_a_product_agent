# 第四章：运维架构选型

> **核心论点**：高并发 C 端服务的可用性靠的不是"买好机器"，而是"没有单点故障"。本章从前后端分离、反向代理、主从架构、哨兵模式、机器选型五个维度，讲清楚生产环境的容灾设计。

---

## 4.1 为什么要前后端分离

### 4.1.1 什么是前后端分离？

传统单体架构中，后端直接把 HTML 模板渲染好返回给浏览器。前后端分离后：

```
               ┌──────────────────────┐
               │    Nginx (反向代理)    │
               └──────┬───────────────┘
                      │
        ┌─────────────┼─────────────┐
        │             │             │
        ▼             ▼             ▼
   /api/*         /assets/*      /* (其他)
        │             │             │
        ▼             ▼             ▼
   ┌─────────┐  ┌─────────┐  ┌──────────┐
   │ Backend │  │  CDN    │  │ Frontend │
   │ :8000   │  │ (静态)   │  │ :3000    │
   │ FastAPI │  │         │  │ React    │
   └─────────┘  └─────────┘  └──────────┘
```

后端只负责提供 API 接口（返回 JSON），前端用 React/Vue 等独立开发、独立部署，通过 HTTP 请求调用后端。

### 4.1.2 为什么必须分离？

**理由一：独立扩容。**

C 端场景下，前端的访问压力和后端的计算压力不总是同比例增长。一个营销活动带来大量浏览（前端压力大，Agent 调用少），而客服高峰期（后端 Agent 调用密集，前端加载少）。分离后可以各扩各的：

```
后端压力大？只加后端节点：
  后端 × 5 + 前端 × 1 ← 合理

前端流量大？只加前端静态节点：
  后端 × 1 + 前端 × 3 ← 合理
```

不分离的架构只能"整体加机器"，浪费严重。

**理由二：独立部署，互不卡发布。**

前端改个 UI 样式不需要后端重新部署。后端升级 Agent 逻辑不影响前端在线的用户。C 端产品迭代快，一天可能发多个版本，绑定部署意味着每次发布都是"两端的代码一起测试、一起回归"——效率极低。

**理由三：职责专一，故障隔离。**

前端服务器挂了，不影响已经在对话中的用户（长连接连的是后端）；后端挂了，前端还能展示降级页面和错误提示，而不是整个白屏。

**理由四：CDN 加速静态资源。**

前端的 HTML / JS / CSS 是纯静态文件，可以推到 CDN 节点上，全国甚至全球用户就近加载。后端 API 返回的动态数据是不能 CDN 缓存的。如果前后端不分离，静态资源和动态请求混在一起，CDN 无法利用。

## 4.2 Nginx 是什么？

### 4.2.1 一句话定义

**Nginx 是一个高性能的反向代理服务器。** 它在用户和后端服务之间充当"接线员"——用户不和后端直接通信，而是通过 Nginx 中转。

```
用户浏览器 ──▶ Nginx ──▶ 后端服务
                    │
                    ├──▶ 负载均衡（轮询/最少连接/IP hash 分发给多台后端）
                    ├──▶ SSL 终结（Https 证书在 Nginx 层处理）
                    ├──▶ 静态文件服务（前端 HTML/JS/CSS 直接由 Nginx 返回）
                    ├──▶ 限流（限制单个 IP 的请求频率）
                    └──▶ 缓存（对部分 API 响应做短时缓存）
```

### 4.2.2 Nginx 在这个系统中的五个具体职责

| 职责               | 说明                          | 示例                                                            |
| ------------------ | ----------------------------- | --------------------------------------------------------------- |
| **反向代理** | 把外部请求转发到内部后端端口  | `proxy_pass http://backend:8000`                              |
| **负载均衡** | 把请求分发给多台后端          | `upstream backend { server b1:8000; server b2:8000; }`        |
| **SSL 终结** | 在这里处理 HTTPS，后端用 HTTP | 证书配在 Nginx 上，后端无需关心                                 |
| **静态资源** | 直出前端文件，不需要经后端    | `root /usr/share/nginx/html;`                                 |
| **限流**     | 保护后端不被刷爆              | `limit_req_zone $binary_remote_addr zone=api:10m rate=30r/s;` |

### 4.2.3 为什么不用后端直接暴露？

```
❌ 错误姿势：
  用户 ──HTTPS──▶ FastAPI:8000

✅ 正确姿势：
  用户 ──HTTPS──▶ Nginx:443 ──HTTP──▶ FastAPI:8000 (内网)
```

- FastAPI 的 HTTPS 性能远不如 Nginx（后者用 C 写的，专门优化过 SSL 处理）
- 后端挂了，Nginx 可以返回一个友好的 502 页面而不是无响应
- 多后端时，客户端（浏览器）不知道有多个后端实例，只跟 Nginx 打交道
- Nginx 可以在一处做限流、日志、gzip 压缩，后端不用每个人都写一遍

## 4.3 生产节点为什么至少 2 个？

### 4.3.1 概念：单点故障（SPOF）

> 单点故障 = 一个组件挂了，整个系统就不可用。

一台机器无论配置多高、买的多年，它都可能在任意时刻挂掉——硬件故障、内核 panic、机房断电、误操作 `kill` 错进程。

### 4.3.2 2 节点是最小高可用单元

```
           ┌──────────────┐
           │    Nginx      │
           └──────┬───────┘
                  │
        ┌─────────┼─────────┐
        │         │         │
        ▼         │         ▼
  ┌──────────┐   │   ┌──────────┐
  │ Backend 1│   │   │ Backend 2│
  │  (正常)   │   │   │  (正常)  │
  └──────────┘   │   └──────────┘
        │         │         │
        ▼ 挂了     │         ▼
  ┌──────────┐   │   ┌──────────┐
  │ Backend 1│   │   │ Backend 2│
  │  (挂了)   │   │   │  (全部请求 │
  │          │   │   │   接管)    │
  └──────────┘   │   └──────────┘
                 │
        Nginx 健康检查检测到 Backend 1 不可用
        → 自动将流量全部转发给 Backend 2
        → 用户无感知
```

**2 个节点 = 挂 1 个还剩 1 个能撑住。** 这就是"冗余"的概念：N+1 冗余（你有 2 台就能承受 1 台挂）。

对于 C 端服务，**高可用 > 高性能**。用户宁可等 3 秒（一台机器满载），也不愿意看到"服务不可用"。

### 4.3.3 3 节点更安全

2 节点的风险：挂了一台之后只剩 1 台，此时再挂就全完了。现实中机器故障往往是相关的（同一批硬件、同一个机柜），所以生产系统推荐至少 3 个节点，确保"挂一台后还剩 N≥2"。

## 4.4 数据库主从架构

### 4.4.1 什么是主从复制？

PostgreSQL 的主从复制是指：一台主机（Primary）负责处理写请求，一台或多台备机（Standby）持续同步主机的数据。写走主、读可走从：

```
         ┌──────────────┐
         │   Primary     │  ← 所有写操作（INSERT/UPDATE/DELETE）
         │   PostgreSQL  │
         └──────┬───────┘
                │ WAL 日志流式同步
       ┌────────┼────────┐
       │        │        │
       ▼        ▼        ▼
 ┌─────────┐┌─────────┐┌─────────┐
 │ Standby 1││ Standby 2││ Standby 3│  ← 分担读操作（SELECT）
 └─────────┘└─────────┘└─────────┘
```

### 4.4.2 为什么需要主从架构？

**理由一：读写分离，扛并发。**

Agent 平台的读写比例大约是 **读 80% : 写 20%**（读对话历史、读知识库、读配置是常态；写对话记录、写日志是间歇性）。主从架构让所有 SELECT 请求分散到从库，Primary 专心处理写入，整体吞吐量成倍提升。

**理由二：高可用——主机挂了切备机。**

```
Primary 宕机
   → 提升一台 Standby 为新 Primary
   → 应用层连接切换到新 Primary
   → 服务恢复（通常在 30 秒内完成）
```

**理由三：备份不锁表。**

传统做法是在主库直接跑 `pg_dump` 备份——备份期间表被锁，写入阻塞。主从架构下，在 Standby 上跑备份，主库完全不感知。

### 4.4.3 主从 vs 单体

| 维度     | 单体 PostgreSQL          | 主从架构               |
| -------- | ------------------------ | ---------------------- |
| 故障恢复 | 手动恢复，30min ~ 数小时 | 自动切换，< 30s        |
| 读吞吐   | 单机上限                 | 水平扩展（加从库）     |
| 备份影响 | 备份时可能锁表           | 从库备份，零影响       |
| 成本     | 1 台机器                 | 至少 2 台（通常 3 台） |

## 4.5 Redis 哨兵模式

### 4.5.1 Redis 三种部署模式对比

```
① 单点模式                    ② 哨兵模式                    ③ 集群模式
┌─────────┐              ┌──────────────┐           ┌──────────────┐
│  Redis  │              │   Sentinel   │           │ Redis Cluster │
│ 单实例   │              │   监控集群    │           │  多主多从     │
│         │              │   ┌───┬───┐  │           │ ┌──┬──┬──┬──┐ │
│ 挂了=全完│              │   │ S1│ S2│  │           │ │M1│M2│M3│M4│ │
└─────────┘              │   └───┴───┘  │           │ └──┴──┴──┴──┘ │
                         │   ┌───┬───┐  │           │  每个M有从节点  │
                         │   │ M │ S │  │           │ 数据分片存储    │
                         │   └───┴───┘  │           └──────────────┘
                         └──────────────┘
```

### 4.5.2 为什么选哨兵模式？

**单点模式的致命缺陷：**

Redis 是存内存数据的（缓存、限流计数、分布式锁），一挂数据全丢。虽然 Redis 支持 RDB/AOF 持久化，但单点模式下恢复的也是上次持久化那一刻的数据——缓存丢得起（重建即可），但**限流计数和分布式锁状态丢了会导致逻辑错误**。而且单点挂了 → 缓存全 Miss → PostgreSQL 瞬间被压垮（缓存雪崩）。

**集群模式过重：**

Redis Cluster 解决的是**数据量大到单机内存装不下**的问题（比如几十 GB 的缓存数据），它会把数据分片（sharding）存到多台机器上。但 Agent 平台的 Redis 数据量通常不大（对话缓存 ≤ 几 GB，计数器 ≤ 几十 MB），你不需要分片，你只需要**高可用**。

**哨兵模式刚好：**

- 一套哨兵进程（Sentinel）持续监控主 Redis 是否存活
- 主挂了 → 自动选举一个从节点提升为新主
- 应用层通过哨兵获取当前主节点地址，自动切换
- 部署简单，运维理解成本低

```
          Sentinels (最少3个，做多数派投票)
          ┌───┐  ┌───┐  ┌───┐
          │S1 │  │S2 │  │S3 │    ← 监控 + 投票
          └───┘  └───┘  └───┘
              \    │    /
               \   │   /
            ┌───▼───▼───▼───┐
            │   Redis Master │  ← 读写
            └───────┬───────┘
                    │ 异步复制
            ┌───────▼───────┐
            │  Redis Slave  │  ← 只读 + 随时准备提升为主
            └───────────────┘
```

### 4.5.4 选型决策表

| 场景                      | 推荐模式       | 理由               |
| ------------------------- | -------------- | ------------------ |
| 开发/DEMO                 | 单点           | 简单，挂了重启就行 |
| 生产（数据量 < 单机内存） | **哨兵** | 高可用，无分片需求 |
| 生产（数据量 > 单机内存） | 集群           | 需要数据分片       |
| 超大规模（TB 级缓存）     | 集群 + 哨兵    | 两者结合           |

Agent 平台的 Redis 属于"数据量不大但绝对不能挂"——哨兵模式正合适。

## 4.6 怎么评估生产需要的机器配置

### 4.6.1 为什么用 Linux？

**不是 Windows 不能跑，而是生产环境没人用 Windows 跑服务端。**

- **进程模型差异**：Linux 的 epoll 在高并发网络 I/O 上远优于 Windows 的 IOCP（uvicorn 的 uvloop 基于 libuv，底层就是 epoll）
- **容器化**：Docker/K8s 的底层是 Linux namespace + cgroup，Windows 跑容器本质是在虚拟机里跑 Linux
- **运维工具链**：systemd 服务管理、iptables/nftables 防火墙、cron 定时任务、ssh 远程管理——这些在 Linux 上是标配
- **生态一致性**：你的开发机、CI/CD 环境、测试环境、生产环境全部用 Linux，不会有"我 Windows 上跑得好好的，上生产就炸了"的问题
- **许可证成本**：Linux 免费，Windows Server 需要购买 CAL 授权

### 4.6.2 推荐哪个 Linux 发行版？

| 发行版                               | 适用场景                 | 推荐度             |
| ------------------------------------ | ------------------------ | ------------------ |
| **Ubuntu Server LTS**（24.04） | 通用生产首选             | ★★★★★         |
| Debian Stable                        | 追求极致稳定             | ★★★★           |
| Rocky Linux / AlmaLinux              | CentOS 替代，企业风      | ★★★★           |
| Alpine                               | 容器镜像专用（极小体积） | ★★★（仅容器内） |

**推荐 Ubuntu Server 24.04 LTS**：生态最大、网上资料最多、各种预编译包最全、LangChain 和 FastAPI 的官方测试环境都在 Ubuntu 上跑。

### 4.6.3 生产节点资源标准怎么算

以 C 端 Agent 平台为例，按**单节点承载能力**倒推：

```
假设：
  - 日活用户 10 万
  - 高峰并发（同时在线） 3000 人
  - 每个 Agent 请求耗时 5~20 秒
  - 每个请求内存占用 ~200MB（加载模型上下文 + Agent 状态）

计算后端节点：
  并发 3000 × 200MB = 600GB 总内存需求
  如果每节点 16GB 内存 → 约 40 个后端节点（这是按瞬时全并发算的极端情况）

  实际：同时"正在 LLM 推理中"的请求远小于并发在线数，通常 10~20%
  真实并发推理数 ≈ 3000 × 15% = 450
  450 × 200MB ≈ 90GB → 约 6 台 16G 节点（加冗余 ≈ 8~10 台）
```

但这是**全链路计算型**的配置。大多数服务的瓶颈不在 CPU/内存，而在 I/O（数据库、网络）。所以主流通用配置是：

### 4.6.4 为什么市面上大量服务是 4C8G200G？

```
┌─────────────────────────────────────────┐
│  4C8G200G 云服务器                       │
│  ┌─────────────────────────────────┐    │
│  │ CPU:  4 核（vCPU × 4）           │    │
│  │ 内存:  8 GB                     │    │
│  │ 系统盘: 40 GB (SSD)             │    │
│  │ 数据盘: 200 GB (SSD)            │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

这是云厂商的"甜蜜点"配置，几乎每个云厂商在促销时主推的规格：

**CPU 4 核 —— 为什么不是 2 核也不是 8 核？**

- 2 核：跑一个 FastAPI（4 workers）+ 一个 Nginx + 一些系统进程就吃满了，留给业务的计算能力几乎没有
- 4 核：FastAPI 开 4 个 worker（每核一个），CPU 利用率刚好 70%左右 且有冗余空间，不会浪费，也能扛住一定并发
- 8 核：对大多业务浪费——你 CPU 跑满了的时候通常是内存或 IO 先爆了
- **4 核是 Web 服务的最优性价比拐点**，再多核不如多加节点（横向扩展）

**内存 8 GB —— 为什么不是 4G 也不是 16G？**

- 4 GB：操作系统自身吃 ~1G，FastAPI worker × 4 基础占用 ~500MB，只剩 ~2.5G 给业务数据——Agent 上下文 + Python 对象很容易就撑爆
- 8 GB：操作系统 ~1G + 应用 ~2G + Agent 运行时 ~3G，剩余 ~2G 做文件缓存和缓冲，不富裕但够用
- 16 GB：如果单个 Agent 上下文特别大或要本地加载模型，才需要。通用 API 服务用不到

**数据盘 200 GB —— 为什么不是 40G 也不是 500G？**

- 40 GB 系统盘装完系统 + Python 环境 + 依赖包后剩 ~15G，日志一写就满
- 200 GB：日志（滚动保留 30 天 ~50GB）+ Docker 镜像 ~20G + 临时文件 + 数据库备份临时目录 = 够用且不浪费
- 500 GB：除非你要在本地存大量文件（用户上传、模型文件），否则用不上

**一句话总结：4C8G200G 是"不浪费、不卡脖子"的配置点。** 这个配置跑一个 FastAPI + Agent 的节点，轻松承载 200~500 QPS（取决于 Agent 复杂度）。

### 4.6.5 平台各组件推荐配置

| 角色               | 配置      | 数量 | 说明                                          |
| ------------------ | --------- | ---- | --------------------------------------------- |
| Nginx              | 2C4G      | 2    | 纯转发，CPU 密集度低                          |
| Backend (FastAPI)  | 4C8G200G  | ≥ 2 | 业务核心，按并发线性扩容                      |
| Agent Worker       | 8C16G200G | ≥ 2 | Agent 执行更吃内存和 CPU                      |
| PostgreSQL Primary | 8C16G500G | 1    | 写入瓶颈在磁盘 IO，需高 IOPS SSD              |
| PostgreSQL Standby | 4C8G200G  | 2    | 只读，配置可低于主库                          |
| Redis + Sentinel   | 2C16G100G | 3    | 单线程模型 CPU 够用即可；内存给足保缓存命中率 |
| RabbitMQ           | 4C8G200G  | 3    | 集群 + 镜像队列                               |
| Milvus             | 8C32G500G | ≥ 3 | 向量索引吃内存和 CPU                          |

## 4.7 DEMO vs 生产的现实落差

开发 DEMO 的时候：

```
┌─────────────────────────────────────────┐
│  你的开发机（或者一台 4C8G 云服务器）       │
│                                          │
│  docker compose up -d                    │
│    ├── PostgreSQL (单实例)               │
│    ├── Redis (单点)                      │
│    ├── Milvus (standalone 模式)          │
│    └── FastAPI (直接跑，不用 Nginx)       │
│                                          │
│  全部在一台机器上，内存勉强够用            │
│  挂了就重启，没有高可用需求                │
└─────────────────────────────────────────┘
```

**DEMO 阶段不需要搭建主从、哨兵、多节点——那是浪费时间和钱。** 但你写系统设计书的时候，必须把本章的内容转化为"生产环境部署架构图"和"机器资源估算表"，让评审人知道：你理解生产环境的复杂度，只是当前阶段不需要。

## 4.8 本章小结

| 选型             | 核心原因                                           |
| ---------------- | -------------------------------------------------- |
| 前后端分离       | 独立扩容、独立部署、CDN 加速                       |
| Nginx 反向代理   | 负载均衡 + SSL + 限流 + 静态资源，后端不需要重复造 |
