# 第三章：协议与架构模式选型

> **核心论点**：协议决定"对外的兼容性"，架构模式决定"对内的执行质量"。本章逐一拆解 OpenAI 兼容协议、ReAct、A2A、Plan-and-Execute、Human-in-the-Loop、Workflow 六个关键选型。

---

## 3.1 OpenAI 兼容协议

### 3.1.1 什么是 OpenAI 兼容协议？

OpenAI 在发布 ChatGPT API 时定义了一套 REST API 规范，其核心端点和数据格式被全行业采纳为事实标准：

```
POST /v1/chat/completions
Content-Type: application/json

{
  "model": "gpt-4o",
  "messages": [
    {"role": "system", "content": "你是一个客服助手"},
    {"role": "user", "content": "帮我查订单"}
  ],
  "stream": true,
  "temperature": 0.7,
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "search_order",
        "description": "查询用户订单",
        "parameters": { "type": "object", "properties": {...} }
      }
    }
  ]
}
```

所谓"兼容 OpenAI 协议"，就是你的 API 也按这套格式收发数据——请求体结构、响应体结构、SSE 事件格式、tool_choice 机制，全部对齐。

### 3.1.2 为什么一定要用它？

**理由一：生态兼容，换个模型不用改代码。**

你作为 Agent 平台，前端和调用方可能是其他团队的开发者。如果你的 API 是自创格式，每个接入方都要单独写适配层。用 OpenAI 兼容协议，接入方只需改个 `base_url`，其他代码不变：

```python
# 接入你的平台，只需改 base_url
client = OpenAI(
    base_url="https://your-agent-platform.com/v1",  # ← 只改这一行
    api_key="your-key"
)
# 其余代码全不变
response = client.chat.completions.create(
    model="your-agent-model",
    messages=[...],
    stream=True
)
```

**理由二：全生态工具链即插即用。**

市面上几乎所有 LLM 工具链、调试工具、监控平台都支持 OpenAI 协议：
- 前端聊天组件（如 Vercel AI SDK）直接对接
- 可观测性平台（Langfuse、LangSmith）能解析标准格式
- 本地开发工具（Open WebUI、ChatBox）零配置接入
- 压测工具（如 locust + openai plugin）拿来就用

你不需要为每个工具写适配。

**理由三：LangChain / LangGraph 原生适配。**

LangChain 的 `ChatOpenAI` 类支持通过 `base_url` 指向任意 OpenAI 兼容服务。你的 Agent 平台对外暴露该协议后，内部微服务之间也可以互相调用，共享同一套客户端代码。

### 3.1.3 向外暴露 OpenAI 兼容 API 的示意图

```
外部调用方（前端 / 第三方）
        │
        │ POST /v1/chat/completions  （OpenAI 标准格式）
        ▼
┌──────────────────┐
│  你的 FastAPI     │
│  /v1/chat/       │
│  completions      │
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│  LangGraph Agent  │  ← 内部执行，结果按 OpenAI 格式返回
└──────────────────┘
```

## 3.2 ReAct 架构（Reasoning + Acting）

### 3.2.1 什么是 ReAct？

ReAct 是 **Reasoning（推理）+ Acting（行动）** 的缩写，是当前智能体架构的核心范式。它的运行方式是一个循环：

```
       ┌──────────────────────┐
       │     LLM 推理          │
       │ "用户想查订单，        │
       │  我需要调用工具..."    │
       └──────────┬───────────┘
                  │
                  ▼
       ┌──────────────────────┐
       │     工具执行           │
       │ 调用 search_order()   │
       │ 返回结果              │
       └──────────┬───────────┘
                  │
                  ▼
       ┌──────────────────────┐
       │     LLM 再推理        │
       │ "查到3个订单，         │
       │  现在生成回复..."     │
       └──────────┬───────────┘
                  │
                  ▼
              （循环直到完成）
```

每一步 LLM 先"想"（推理是否要调工具、调哪个工具、参数是什么），再"做"（执行工具调用），然后根据工具返回结果再"想"下一步——这就是 ReAct。

### 3.2.2 为什么 Agent 必须用 ReAct？

传统 LLM 调用是"一问一答"：用户问、模型答，中间没有工具调用环节。Agent 的本质区别在于**能把外部信息拉进来**——查数据库、调 API、搜文档——这些能力需要 LLM 在对话中间"暂停"去调工具。

ReAct 就是这个"暂停-调工具-接着想"的循环机制。没有 ReAct，Agent 就是普通聊天机器人。

### 3.2.3 ReAct 在 LangGraph 中的体现

LangGraph 的 `ToolNode` 和条件边就是 ReAct 的工程实现：

```python
# LangGraph 中的 ReAct 循环
graph_builder.add_node("agent", agent_node)     # Reasoning：LLM 判断
graph_builder.add_node("tools", tool_node)       # Acting：执行工具
graph_builder.add_conditional_edges(
    "agent",
    router,           # 有 tool_calls → 去 tools，否则 → END
    {"tools": "tools", "__end__": "__end__"}
)
graph_builder.add_edge("tools", "agent")         # 工具执行完 → 回到 agent 再推理
```

这就是一个标准的 ReAct 循环图。

## 3.3 A2A（Agent-to-Agent）协议

### 3.3.1 什么是 A2A？

**A2A（Agent-to-Agent）** 是 Google 于 2025 年发布的开放协议，专门解决"一个 Agent 怎么调用另一个 Agent"的问题。它定义了：

- **Agent Card**：一个 JSON 描述文件（`/.well-known/agent.json`），列出该 Agent 的能力、端点、认证方式
- **Task 生命周期**：每个任务有唯一 ID，状态包括 submitted → working → completed / failed
- **多模态传递**：支持文本 + 文件 + 结构化数据在 Agent 之间传递
- **流式传输**：支持 SSE 流式推送结果

### 3.3.2 为什么多 Agent 通信不能只用 HTTP + JSON？

因为 Agent 之间的调用和普通微服务调用有本质区别：

| 维度 | 普通微服务调用 | Agent 间调用 |
|------|---------------|-------------|
| 响应时间 | < 100ms | 几秒到几十秒 |
| 通信模式 | 一问一答 | 多轮推理 + 中间工具调用 |
| 状态追踪 | 无需 | 需要知道"Agent B 正在执行到哪一步" |
| 能力发现 | 写死在代码里 | 需要动态发现"这个 Agent 能做什么" |
| 取消/中断 | HTTP 断开即可 | 需要有语义的取消（如"撤销部署任务"） |

A2A 协议把这些 Agent 特有的需求标准化了。

### 3.3.3 A2A 在这个平台中的场景

```
用户："帮我写一份 Q2 销售分析报告"

   用户 Agent（前台）
      │
      ├──▶ A2A 调用：数据 Agent
      │    "查询 Q2 各地区销售额 TOP10" → 返回结构化数据
      │
      ├──▶ A2A 调用：文档 Agent  
      │    "检索销售分析报告模板" → 返回模板内容
      │
      └──▶ 汇总信息 → 调用 LLM 生成最终报告
```

用户 Agent 自己不查数据库、不搜文档——它的能力边界是理解意图和生成文本。数据和文档是另外两个 Agent 的专业领域，通过 A2A 协议各司其职。

### 3.3.4 什么时候不需要 A2A？

- 只有 1 个 Agent（不需要"之间"通信）
- Agent 逻辑简单，用 LangGraph 的 subgraph 就够了（子图本质是函数调用，不是独立服务）
- DEMO 阶段直接用 HTTP + JSON 即可，A2A 是生产规模化的需要

## 3.4 Plan-and-Execute（先规划再执行）

### 3.4.1 什么是 Plan-and-Execute？

普通的 ReAct 模式是**一步一想、一步一做**——LLM 每次都只决定"下一步做什么"，缺乏全局视角。这容易导致：
- 执行到第 3 步才发现方向错了
- 复杂任务迷失在中间步骤里
- 用户无法提前知晓"总共要做几步"

Plan-and-Execute 的改进是：**先让 LLM 生成一个完整的执行计划，用户确认后，再逐步执行。**

```
用户： "帮我分析竞品 A 的定价策略"

【Plan 阶段】
  1. 调用 search_web("竞品A 价格") 获取公开价格信息
  2. 调用 db_query("SELECT price FROM our_products") 获取我方价格
  3. 调用 knowledge_base("定价分析框架") 获取分析方法论
  4. 综合分析 → 生成报告

  ┌──────────────┐
  │ 展示计划给用户 │ → 用户确认 / 修改 / 取消
  └──────┬───────┘
         │
【Execute 阶段】
  按计划逐步执行，每步汇报进度
```

### 3.4.2 为什么需要它？

**对用户的价值：**
- 知道自己要等多久、Agent 要做什么、要不要干预
- 复杂任务用户可能有偏好（"别用竞品 A 的价格来源？那就换数据源"），plan 阶段就可以调整

**对系统的价值：**
- Plan 可以作为审计日志的基础颗粒度
- 出现错误可以精确定位到"计划的第 X 步失败"
- 部分步骤可并行（如步骤 1 和 2 互不依赖，同时执行）

### 3.4.3 实现思路（LangGraph 中的子图）

```
┌──────────────────────────────────────┐
│         Plan and Execute Graph         │
│                                      │
│  ┌─────────┐     ┌──────────────┐   │
│  │ Planner  │────▶│ Human Review │   │
│  │ (生成计划)│     │ (确认/修改)  │   │
│  └─────────┘     └──────┬───────┘   │
│                         │           │
│                         ▼           │
│                  ┌──────────────┐   │
│                  │  Executor     │   │
│                  │  逐步执行      │   │
│                  │  (子 ReAct 图) │   │
│                  └──────┬───────┘   │
│                         │           │
│                         ▼           │
│                  ┌──────────────┐   │
│                  │  Reporter     │   │
│                  │  (汇总报告)    │   │
│                  └──────────────┘   │
└──────────────────────────────────────┘
```

Planner 是一个专门的 LLM 调用节点，输出结构化的步骤列表。Executor 是另一个 ReAct 子图，逐步骤执行。

## 3.5 Human-in-the-Loop（人机协同）

### 3.5.1 什么是 Human-in-the-Loop？

Human-in-the-Loop（HITL）是指 Agent 在**关键节点暂停执行，等待人工审批或输入**后才能继续。在 LangGraph 中通过 `interrupt` 机制实现。

```
Agent 执行流程:
  推理 → 查询数据 → 生成回复草案 ──▶ 【暂停，等待人工审核】 ──▶ 发送给用户
                                        │
                                   ┌────▼────┐
                                   │ 运营人员  │
                                   │ 审核/修改  │
                                   └────┬────┘
                                        │
                                   ┌────▼────┐
                                   │ 批准发送  │
                                   └─────────┘
```

### 3.5.2 为什么必须有它？

**场景一：外部 C 端客服 Agent。**

Agent 可以自动回复 90% 的常规问题，但当它检测到以下情况时必须暂停：
- 用户情绪明显愤怒（"你们这破产品！"）
- Agent 的置信度低于阈值
- 涉及退款、赔偿等敏感操作

自动回复"您的退款已处理"但实际没处理——这是生产事故。

**场景二：内部代码审查 Agent。**

Agent 生成代码后，必须经人审批才能推送。AI 写的代码可能有安全漏洞，人审一下成本极低，不审的代价可能很大。

**场景三：数据分析 Agent。**

分析维度可能偏差，在实际产出报告之前让数据科学家确认计划，比产出后推翻重做高效得多。

### 3.5.3 LangGraph 的 `interrupt` 机制

```python
from langgraph.checkpoint.memory import MemorySaver
from langgraph.types import interrupt

def sensitive_action_node(state):
    draft = generate_draft(state)
    # 暂停，把草案推给人工审核终端
    approved = interrupt({"type": "review_required", "draft": draft})
    # 人工审批通过后，interrupt 返回审批结果，继续执行
    if approved.get("decision") == "approved":
        return execute(draft)
    return discard(draft)

# 需要配置 checkpointer 才能使用 interrupt
graph = builder.compile(checkpointer=MemorySaver())
```

## 3.6 Workflow（工作流）

### 3.6.1 什么是 Workflow？

Workflow 是**预定义好步骤序列的确定性流程**。每一步做什么固定，不依赖 LLM 的"判断"。对比：

| | ReAct Agent | Workflow |
|------|-------------|----------|
| 流程决定者 | LLM 动态决定 | 开发者提前定义 |
| 确定性 | 不确定（同一输入可能不同路径） | 确定（输入 A 必定走步骤 1→2→3） |
| 适用场景 | 开放域对话、复杂推理 | 表单处理、审批流、数据 ETL |
| 代价 | LLM 调用费用 + 延迟 | 几乎零 LLM 调用 |

### 3.6.2 为什么有 ReAct 还需要 Workflow？

因为现实业务中，**不是所有问题都需要 LLM 思考**。

**场景一：用户想重置密码。**

这不是需要"推理"的事。Workflow 就三步：

```
输入: 用户邮箱
  → Step 1: 验证邮箱格式
  → Step 2: 生成重置 token 存入数据库  
  → Step 3: 发送重置邮件
输出: "密码重置邮件已发送"
```

每一步都是确定的代码逻辑，不需要 LLM 参与。用 Agent 做这个反而慢且浪费钱。

**场景二：多步骤数据采集。**

用户要创建一个复杂的业务配置（比如配置一个监控告警规则），需要依次填写：
1. 选择监控指标
2. 设置阈值
3. 配置通知渠道
4. 确认并保存

这是一个多步表单，Workflow 精确控制每个步骤的跳转和校验。Agent 无法替代。

**场景三：审批流。**

一个报销单走"员工提交 → 主管审批 → 财务复核 → 出纳打款"，每一步的流转规则是确定的。Workflow 精确控制，不需要 AI 决策。

### 3.6.3 工程上的互补关系

```
用户请求进入
   │
   ▼
┌──────────────┐
│   Router      │  ← 意图识别（可以用 LLM 做）
└──┬───────┬───┘
   │       │
   ▼       ▼
┌──────┐ ┌────────┐
│Workflow│ │ Agent  │
│确定流程│ │智能推理│
│(快/便宜) │(灵活/强)│
└──────┘ └────────┘
```

两者不是替代关系，是互补关系。一个成熟的 Agent 平台应该：
- 能用 Workflow 解决的，坚决不调 Agent（省钱、快、可靠）
- Workflow 兜不住的（开放对话、复杂推理、多工具编排），才走 Agent

这就是 **Router-Workflow-Agent 三层架构** 的核心思想。

## 3.7 六个选型关系总结

```
                        ┌──────────────────────┐
                        │   OpenAI 兼容协议     │
                        │   (对外统一的交互语言) │
                        └──────────┬───────────┘
                                   │
              ┌────────────────────┼────────────────────┐
              │                    │                    │
              ▼                    ▼                    ▼
     ┌────────────┐     ┌───────────────┐     ┌──────────────┐
     │  Workflow  │     │  ReAct Agent  │     │  A2A 协议    │
     │ (确定流程)  │     │  (智能推理)   │     │ (多Agent通信) │
     │            │     │               │     │              │
     │ 表单/审批  │     │  内部包含：    │     │ Agent间发现  │
     │ ETL/规则   │     │  Plan-and-Execute │     │ 任务追踪     │
     └────────────┘     │  HumanInLoop  │     └──────────────┘
                        └───────────────┘
```

- **OpenAI 兼容协议** 是你这个平台对外的"语言"——一切调用方都用这个格式跟你对话
- **ReAct** 是单个 Agent 内部的核心运行方式——推理 + 行动交替
- **A2A** 是多个 Agent 之间的通信标准——让 Agent 能像微服务一样互相发现和调用
- **Plan-and-Execute** 是 ReAct 之上的增强——加上计划阶段，提升复杂任务的可靠性和可解释性
- **Human-in-the-Loop** 是安全的最后一道防线——关键节点必须人确认才能继续
- **Workflow** 是 Agent 的补充——把确定的流程代码化，避免不必要的 LLM 调用

---

## 3.8 本章小结

六个选型围绕一个核心问题：**"Agent 平台如何对外交互、内部执行、彼此协作？"**

| 选型 | 解决的问题 |
|------|-----------|
| OpenAI 兼容协议 | 对外统一接口，生态即插即用 |
| ReAct | 单个 Agent 的推理-行动循环 |
| A2A | 多 Agent 之间的标准化通信 |
| Plan-and-Execute | 复杂任务的全局规划与可控执行 |
| Human-in-the-Loop | 关键节点的安全兜底 |
| Workflow | 确定性流程的零 AI 成本执行 |
